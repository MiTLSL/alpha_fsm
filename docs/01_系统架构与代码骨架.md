# 01 系统架构与代码骨架

> 本文档是全员开发前必读。它定义了进程划分、代码包结构、FSM 引擎接口、线程模型、父子 FSM 调用模式、Context 与状态广播。所有人按本规范实现，避免边界处冲突。

---

## 0. TL;DR

```text
五层 FSM × 核心 5 个 ROS2 节点 + 2 个适配节点 × 单线程 tick + 关键节点局部多线程
父 FSM 同进程类调用四个策略子 FSM，跨节点 Action 调用导航与抓取执行
StateContext 节点内对象 + 关键状态 Topic 广播给 UI / 复盘
急停走独立 callback group + 主 tick 标志位检查
```

### 1.2 节点划分原则

为什么核心 FSM 是 5 个，而不是 1 个或把 adapter 混进策略节点：

```text
1. 按"故障域"拆分：safety_monitor 必须独立进程，避免策略节点崩溃带不掉急停
2. 按"开发归属"拆分：A、B 各自一个节点，并行开发互不阻塞
3. 按"长耗时动作"拆分：navigation 与 pair_grasp 是 Action server，本来就该独立
4. 同一团队同一职责的策略类 FSM 不再拆：四个策略子 FSM 全部塞进 strategy_node
5. SystemFSM 与 TaskFSM 都属于"任务调度"层，合并为 task_manager_node
6. 感知和真空 IO 属于外部接口适配，不参与业务状态跳转，独立节点便于 mock、替换和健康监控
```

## 3. FSM 引擎核心接口

### 3.1 设计原则

```text
1. 引擎本身只负责"调当前状态、处理跳转、记日志、检查急停"，不知道任何业务
2. 所有业务在 BaseState 子类里
3. 单线程 tick，不引入 asyncio
4. 每次 tick 完整执行一个状态步进，不在状态内部阻塞过久
5. 长耗时动作通过 Action client 发出 goal 后立刻返回 RUNNING，下次 tick 再查 future
```

### 3.2 BaseState 抽象类

```python
# fsm_core/base_state.py
from abc import ABC, abstractmethod
from dataclasses import dataclass
from enum import Enum
from typing import Optional

class TransitionType(Enum):
    STAY = "stay"           # 继续在本状态，下一 tick 再来
    SUCCESS = "success"     # 成功，跳到 on_success_to
    FAILURE = "failure"     # 失败，跳到 on_failure_to（或交给 ErrorHandler）
    TIMEOUT = "timeout"     # 超时
    ABORT = "abort"         # 急停或外部抢占

@dataclass
class StateResult:
    transition: TransitionType
    next_state: Optional[str] = None       # 显式指定下一个状态名（覆盖默认跳转）
    error_code: Optional[int] = None       # 失败/超时时填
    payload: Optional[dict] = None         # 传给下个状态的临时数据

class BaseState(ABC):
    """所有状态的抽象基类。每个具体状态必须继承并实现 execute。"""

    name: str = ""                         # 类属性，状态名（必填）
    fsm_layer: str = ""                    # 类属性，所属 FSM 层级（必填）
    timeout_sec: float = 10.0              # 类属性，默认超时（可被 fsm.yaml 覆盖）
    max_retry: int = 0                     # 类属性，状态级重试次数

    def on_enter(self, ctx) -> None:
        """进入状态时调用一次。可初始化局部变量、记录 enter_time、发起 Action goal。"""
        pass

    @abstractmethod
    def execute(self, ctx) -> StateResult:
        """每个 tick 调用一次。必须在 ~10ms 内返回，不允许阻塞。
        长耗时动作：第一次 tick 发 Action goal，后续 tick 检查 future 并返回 STAY/SUCCESS/FAILURE。"""
        ...

    def on_exit(self, ctx, result: StateResult) -> None:
        """退出状态时调用一次。可清理资源、记录 exit_time。"""
        pass

    def on_timeout(self, ctx) -> StateResult:
        """超时由引擎调用。默认返回 TIMEOUT，状态可重写以做清理（如 cancel goal）。"""
        return StateResult(transition=TransitionType.TIMEOUT,
                           error_code=ErrorCode.E_STATE_TIMEOUT)

    def on_abort(self, ctx, reason: str) -> None:
        """急停或上层抢占时调用，状态必须立即取消正在执行的动作（如 cancel Action goal）。"""
        pass
```

### 3.3 FSMEngine 主循环

```python
# fsm_core/fsm_engine.py
class FSMEngine:
    """单线程 tick 引擎。一个 FSM 一个引擎实例。"""

    def __init__(self, name: str, states: dict[str, BaseState],
                 transitions: dict[tuple[str, TransitionType], str],
                 initial_state: str, ctx, logger, broadcaster):
        self.name = name                  # FSM 名，用于日志和广播
        self.states = states              # 状态名 → BaseState 实例
        self.transitions = transitions    # (from_state, transition) → to_state
        self.ctx = ctx
        self.logger = logger
        self.broadcaster = broadcaster
        self._current = initial_state
        self._enter_time: float = 0.0
        self._abort_flag = False
        self._abort_reason = ""
        self._entered = False

    def tick(self) -> bool:
        """执行一步。返回 True 表示 FSM 还在运行，False 表示已终止。"""
        # 1. 检查抢占
        if self._abort_flag:
            self._handle_abort()
            return False

        state = self.states[self._current]

        # 2. 首次进入处理
        if not self._entered:
            self._enter_time = time.monotonic()
            state.on_enter(self.ctx)
            self.logger.on_enter(self.name, self._current, self.ctx)
            self.broadcaster.publish(self.name, self._current, self.ctx)
            self._entered = True

        # 3. 超时检查
        if time.monotonic() - self._enter_time > state.timeout_sec:
            result = state.on_timeout(self.ctx)
            self._do_transition(state, result)
            return self._current is not None

        # 4. 正常执行一步
        result = state.execute(self.ctx)
        if result.transition == TransitionType.STAY:
            return True

        self._do_transition(state, result)
        return self._current is not None

    def request_abort(self, reason: str) -> None:
        """外部线程调用。下次 tick 会处理。"""
        self._abort_flag = True
        self._abort_reason = reason

    def _do_transition(self, state, result):
        # ... 调 on_exit、记日志、查表跳到下个状态、重置 _entered
        ...

    def _handle_abort(self):
        # ... 调当前状态的 on_abort、记日志、置 _current = None
        ...
```

**关键点**：
- `tick()` 必须 ~10ms 内返回，rclpy 的 `Timer` 用 100Hz 调用即可
- 急停由 `safety_monitor` 节点的 Topic 触发 → callback 调 `engine.request_abort()` → 主 tick 下一次进入会处理
- 不在状态内 sleep；要等的事情用"下一 tick 再查"

### 3.4 父子 FSM 调用模式

#### 同进程类调用（用于策略子 FSM）

父 FSM 的某个状态持有子 FSM 引擎实例，每次 tick 转发：

```python
# 父 FSM 的 RUN_WALL_MAPPING 状态
class RunWallMapping(BaseState):
    name = "RUN_WALL_MAPPING"

    def on_enter(self, ctx):
        self._sub_engine = ctx.wall_mapping_engine     # 节点构造时注入的子 FSM 引擎
        self._sub_engine.reset()                       # 子 FSM 从初始状态重新开始

    def execute(self, ctx) -> StateResult:
        running = self._sub_engine.tick()              # 把这次 tick 转发给子 FSM
        if running:
            return StateResult(TransitionType.STAY)

        sub_result = self._sub_engine.last_result()
        if sub_result.transition == TransitionType.SUCCESS:
            return StateResult(TransitionType.SUCCESS)
        return StateResult(TransitionType.FAILURE,
                           error_code=sub_result.error_code)
```

**关键点**：
- 父子共享 **同一个 ctx**，子 FSM 直接读写 ctx 的字段
- 父 FSM 一个 tick 触发一次子 FSM 的 tick，整体仍是单线程
- 子 FSM 完成后通过 `last_result()` 把结果回报父 FSM

#### 跨节点 Action 调用（用于 BaseNavigationFSM、PairGraspExecutionFSM）

父 FSM 的某个状态用 Action client 发起远程调用：

```python
# 父 FSM 的 NAVIGATE_TO_PHASE_WORKPOSE 状态
class NavigateToPhaseWorkpose(BaseState):
    name = "NAVIGATE_TO_PHASE_WORKPOSE"
    timeout_sec = 60.0

    def on_enter(self, ctx):
        goal = build_nav_goal(ctx.current_phase, ctx.wall_grid)
        self._send_future = ctx.nav_action_client.send_goal_async(goal)
        self._goal_handle = None
        self._result_future = None

    def execute(self, ctx) -> StateResult:
        if self._goal_handle is None:
            if not self._send_future.done():
                return StateResult(TransitionType.STAY)
            self._goal_handle = self._send_future.result()
            if not self._goal_handle.accepted:
                return StateResult(TransitionType.FAILURE,
                                   error_code=ErrorCode.E_NAV_GOAL_REJECTED)
            self._result_future = self._goal_handle.get_result_async()
            return StateResult(TransitionType.STAY)

        if not self._result_future.done():
            return StateResult(TransitionType.STAY)

        nav_result = self._result_future.result().result
        ctx.last_nav_result = nav_result
        if nav_result.success:
            return StateResult(TransitionType.SUCCESS)
        return StateResult(TransitionType.FAILURE, error_code=nav_result.error_code)

    def on_abort(self, ctx, reason):
        if self._goal_handle is not None:
            self._goal_handle.cancel_goal_async()
```

**关键点**：
- Action 是非阻塞的，`tick()` 通过查 `future.done()` 推进
- 急停时 `on_abort` 必须 cancel goal，否则导航/机械臂会继续动
- Action 节点本身内部用什么模型不管——`navigation_manager_node` 可以是 FSM，也可以直接调底层导航库

## 4. 线程模型

### 4.1 各节点线程策略

| 节点 | 线程模型 | 说明 |
|---|---|---|
| `task_manager_node` | 单线程 SingleThreadedExecutor + 100Hz Timer 跑 tick | TaskFSM 没有强实时需求 |
| `wall_destacking_strategy_node` | 单线程 SingleThreadedExecutor + 100Hz Timer 跑 tick | 全部策略类 FSM 串行 tick |
| `navigation_manager_node` | MultiThreadedExecutor + 2 个 callback group | Action server 一组、状态广播一组 |
| `pair_grasp_execution_node` | MultiThreadedExecutor + 多个 callback group | 左/右臂可能并行执行；vacuum 监控独立组 |
| `safety_monitor_node` | MultiThreadedExecutor + 高优先级 group | 急停信号必须不被其他 callback 阻塞 |

### 4.2 急停机制

```text
硬件急停按钮 / 软件急停命令
   ↓
safety_monitor_node 监听
   ↓
广播 /safety/estop 为 true
   ↓
各 FSM 节点的 SafetyListener callback（独立 callback group）
   ↓
立即调 engine.request_abort("estop")
   ↓
主 tick 下一次循环开头检查 abort_flag
   ↓
当前状态 on_abort()，cancel 所有 Action goal
   ↓
FSM 进入 E_STOP 终态，TaskFSM 上报 SystemFSM
```

**强约束**：任何状态的 `on_abort` 实现必须 ≤ 50ms，且必须 cancel 所有 outstanding 的 Action goal。

### 4.3 100Hz 是上限不是下限

- 策略 FSM 节点 tick rate 设为 20Hz 即可（fsm.yaml 可调），减少 CPU 占用
- 急停响应时间 = `1 / tick_rate`，20Hz 时最慢 50ms 响应，可接受
- 如果未来需要更快响应，再调高 tick_rate

## 5. StateContext 设计

### 5.1 StateContext 是什么

`StateContext` 是一个 FSM 节点内部的"工作内存"，由该节点的所有状态共享读写。**它不是分布式 WorldModel**，每个节点维护自己的 ctx。

跨节点的数据同步通过 Action goal/result 字段或 Topic 完成，绝不通过共享 ctx。

### 5.2 task_manager_node 的 TaskContext

```python
@dataclass
class TaskContext:
    # —— 任务元数据 ——
    task_id: str = ""
    task_start_time: float = 0.0

    # —— 系统层状态 ——
    system_mode: str = "BOOTING"              # BOOTING / STANDBY / AUTO / MANUAL / FAULT / E_STOP
    safety_status: SafetyStatus | None = None # 来自 /safety/status

    # —— 任务层状态 ——
    task_state: str = "WAIT_TASK"
    pause_requested: bool = False
    cancel_requested: bool = False

    # —— 子 FSM 调用句柄 ——
    wall_destacking_action_client: Any = None
    wall_destacking_goal_handle: Any = None
    wall_destacking_result: Any = None

    # —— 错误 ——
    last_error: ErrorReport | None = None
    error_history: list[ErrorReport] = field(default_factory=list)
```

### 5.3 wall_destacking_strategy_node 的 WallDestackingContext

```python
@dataclass
class WallDestackingContext:
    # —— 任务元数据（来自 TaskFSM 的 Action goal）——
    task_id: str = ""
    wall_index: int = 0

    # —— Wall 与网格 ——
    current_wall_grid: WallGrid | None = None
    new_wall_detected: bool = False
    empty_confirm_count: int = 0

    # —— Phase ——
    current_phase: str = "LEFT_PHASE"         # LEFT_PHASE / RIGHT_PHASE

    # —— 局部感知 ——
    last_local_detections: list[BoxDetection] = field(default_factory=list)
    visible_slot_ids: list[str] = field(default_factory=list)

    # —— 当前抓取对 ——
    current_grasp_pair: GraspPair | None = None
    last_pair_grasp_result: PairGraspResult | None = None

    # —— 重试与异常 ——
    pair_retry_count: int = 0
    last_error: ErrorReport | None = None

    # —— 同进程子 FSM 引擎 ——
    wall_mapping_engine: FSMEngine | None = None
    phase_perception_engine: FSMEngine | None = None
    pair_selection_engine: FSMEngine | None = None
    wall_recovery_engine: FSMEngine | None = None

    # —— Action client ——
    nav_action_client: Any = None
    pair_grasp_action_client: Any = None

    # —— Service client ——

    # —— 配置参数（从 ROS2 parameters 读出来缓存）——
    config: BusinessConfig | None = None
```

### 5.4 读写规则（强约束）

```text
1. 父 FSM 与同进程子 FSM 共享同一 ctx，可直接读写其字段
2. 跨节点不共享 ctx，必须通过 Action / Service / Topic 传数据
3. ctx 字段全部用 dataclass，禁止用 dict 当 ctx
4. 状态可读 ctx 的任何字段，但写入字段必须在该状态的"输出契约"里声明（见 02 状态规格手册）
5. 不允许把临时变量塞进 ctx；状态内的临时数据用 self._xxx
6. ctx 里的 Action client / Service client 由 node.py 在启动时构造，状态只用不创建
```

### 5.5 状态广播

为了 UI 可视化与日志复盘，每个节点必须对外广播关键状态。**这是方案 C 的核心**。

| Topic | 来自 | 频率 | 内容（详见 03 接口契约） |
|---|---|---|---|
| `/fsm/system_state` | task_manager_node | 状态变化时 + 1Hz 心跳 | system_mode、task_id、task_state |
| `/fsm/task_state` | task_manager_node | 状态变化时 | task_state、pause/cancel flag |
| `/fsm/wall_destacking_state` | strategy_node | 状态变化时 + 1Hz 心跳 | 当前父 FSM 状态、wall_index、phase |
| `/fsm/active_substate` | strategy_node | 状态变化时 | 当前在哪个子 FSM 的哪个状态 |
| `/fsm/grid_snapshot` | strategy_node | 网格变化时 | 5×5 slot 状态全量快照 |
| `/fsm/grasp_pair` | strategy_node | 选好新 pair 时 | 当前 pair 详情 |
| `/fsm/error` | 任何节点 | 出错时 | error_code、level、source、msg |
| `/fsm/log_event` | 任何节点 | 状态跳转时 | 一条 JSON 事件，UI/复盘订阅 |

**实现**：`fsm_core/broadcaster.py` 提供统一的 `Broadcaster` 类，状态跳转时由 FSMEngine 自动调它，用户状态代码不必关心。

## 6. 日志规范

### 6.1 输出形式

**纯 JSON Lines 文件**，路径与字段在 `params/logging.yaml` 配置。每个节点写自己的 `<node_name>_<date>.jsonl`。

不写 ROS2 log，不开 rosbag2（除非现场调试需要再单独开）。

### 6.2 必填字段

每条日志必须有：

```json
{
  "ts": 1717000000.123,
  "node": "wall_destacking_strategy_node",
  "fsm": "WallDestackingFSM",
  "task_id": "task_20260522_001",
  "event": "state_enter | state_exit | transition | error | metric",
  "state": "RUN_PAIR_SELECTION",
  "from_state": "RUN_PHASE_PERCEPTION",
  "to_state": "DISPATCH_PAIR_GRASP",
  "transition": "success",
  "duration_ms": 12.4,
  "error_code": null,
  "retry_count": 0,
  "ctx_snapshot": { "wall_index": 0, "phase": "LEFT_PHASE", "current_grasp_pair_id": "p_0001" },
  "extra": {}
}
```

### 6.3 写入时机（强约束）

```text
state_enter      每次 on_enter 后写一条
state_exit       每次 on_exit 后写一条（含 duration、result）
transition       每次状态跳转写一条（含 from/to/transition_type）
error            任何 error_code 非 None 时写一条
metric           关键耗时打点（感知耗时、规划耗时、抓取耗时等）
```

由 `fsm_core/state_logger.py` 统一封装，FSMEngine 自动调，用户状态不必显式写。

## 7. 配置加载流程

```text
launch 文件
  → 加载 params/business.yaml + fsm.yaml + interfaces.yaml + error_codes.yaml + logging.yaml
  → 注入到对应节点的 ROS2 parameters
  → 节点 __init__ 时 self.declare_parameter() + self.get_parameter()
  → 缓存到 ctx.config 对象（dataclass）
  → 状态代码读 ctx.config.xxx
```

**强约束**：
- 状态代码不直接调 `self.get_parameter()`，统一从 `ctx.config` 读
- `ctx.config` 是不可变快照（@dataclass(frozen=True)）；现场动态调参时需要 reload，由 node.py 提供 `/reload_config` Service

## 8. 启动入口

### 8.1 命令行启动

```bash
# 完整真机启动
ros2 launch fsm_config bringup_full.launch.py

# 用 mock 替换 navigation 和 pair_grasp（开发期常用）
ros2 launch fsm_config bringup_with_mock.launch.py mock_nav:=true mock_grasp:=true

```

### 8.2 任务控制 Service

无论命令行还是 UI，都通过 `task_manager_node` 提供的 Service 控制任务：

```bash
ros2 service call /task/start fsm_msgs/srv/TaskControl "{task_id: 'task_001'}"
ros2 service call /task/pause fsm_msgs/srv/TaskControl "{}"
ros2 service call /task/resume fsm_msgs/srv/TaskControl "{}"
ros2 service call /task/cancel fsm_msgs/srv/TaskControl "{}"
```

UI Web 节点（P2）订阅状态广播 Topic + 调这些 Service 即可，无需修改 FSM 代码。

## 10. 强约束清单（速查）

```text
1. 一个状态一个文件，文件名 = 状态名小写下划线
2. 数据类必须 @dataclass，禁止用 dict
3. 状态 execute() 必须 ~10ms 内返回，禁止 sleep / 阻塞 IO
4. 长耗时动作必须用 Action，不允许用 Service
5. 急停必须能在 50ms 内中断当前状态（on_abort + cancel goal）
6. 跨节点不共享 ctx，必须通过 ROS2 通信
7. 状态读 ctx 任何字段，但写字段必须在状态规格手册里声明
8. 配置统一从 ctx.config 读，不在状态内调 get_parameter
9. 日志由 FSMEngine 自动写，状态代码不显式记日志（除非业务 metric）
10. 状态广播由 FSMEngine 自动发，状态代码不显式 publish 状态
11. 父子 FSM 同进程时共享 ctx；跨节点时通过 Action 字段传
12. 任何 error 必须有 error_code（见 04 错误码文档）
13. tick_rate 默认 20Hz，可在 fsm.yaml 调
```
