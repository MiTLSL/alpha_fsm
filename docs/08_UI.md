# 08 UI 设计

> 本文档定义 FSM 配套的 UI Web 节点（**P2 实现**，本期不阻塞 FSM 主线）。它是 FSM 的**只读监控 + 受限控制面**：状态广播全部从 FSM 节点的 Topic 拉，控制操作通过既有 Service / Action 触发。**UI 不持有任何业务状态**，关掉 UI 不影响任务执行。
>
> 配套阅读：
> - 03 接口契约 §3 / §4（Topic、QoS）
> - 04 错误码（UI 的告警映射）
> - 02 状态规格手册（状态名集合）
> - 06 Mermaid 图集（UI 的状态图直接复用这些）

---

## 0. 设计原则

```text
1. UI 是消费者，不是生产者：状态全部订阅；写操作仅限既有 Service。
2. UI 离线必须不影响任务：所有 Topic 用 RELIABLE+TRANSIENT_LOCAL 让晚启动也能拿到最新；
   Service 是普通的客户端，不绑定 UI 生命周期。
3. UI 不做业务校验：参数合法性、权限、模式约束由 task_manager 决定，UI 只展示拒绝原因。
4. 单页 SPA + WebSocket / rosbridge_server，不引入额外 ROS2 节点（除了 rosbridge）。
5. 颜色与状态名严格对齐 04 错误码 level 与 02 状态规格手册的字符串，不做 UI 端重命名。
6. 急停按钮在 UI 上属于辅助手段，权威是硬件按钮 + safety_monitor。
```

---

## 1. 部署架构

```text
浏览器（Web SPA）
   ↕ WebSocket (rosbridge_server, 9090)
[rosbridge_server, 现成包]
   ↕ ROS 2 (DDS)
[核心 5 个 FSM 节点 + 2 个外部适配节点]
```

- 不写自定义 ROS 节点；rosbridge_server 用上游版本
- SPA 用 React + roslibjs；构建产物静态托管（任意 web 服务器）
- 部署时只需启 `rosbridge_websocket.launch.xml`，再起一个 nginx 指向 `dist/`

> 选 rosbridge 的理由：成熟、消息体直接 JSON、UI 端无需自己实现 DDS。代价是 BEST_EFFORT QoS 下偶尔丢消息，对状态广播无影响（有 1Hz 心跳兜底）。

## 2. 订阅清单（只读）

按 03 §4.1 的推荐清单为底，本节给出 UI 的具体使用：

| `/fsm/task_state` | `FsmStateSnapshot` | 顶栏：任务进度（task_id、task_state、wall_index、phase、elapsed） | 顶栏徽章 |
| `/fsm/wall_destacking_state` | `FsmStateSnapshot` | 中央：当前父 FSM 状态名 + state_elapsed_sec | 状态卡片 |
| `/fsm/active_substate` | `FsmStateSnapshot` | 中央：当前子 FSM 状态名（WallMappingFSM / PhasePerceptionFSM / PairSelectionFSM / WallRecoveryFSM 之一）| 状态卡片下半 |
| `/fsm/grid_snapshot` | `WallGridSnapshot` | 主画布：5×5 网格热力图，每 slot 颜色按 status | 主面板 |
| `/fsm/grasp_pair` | `GraspPair` | 主画布：高亮当前 pair 的左右 slot；侧栏列出 pair 详情 | 主面板叠加 |
| `/fsm/error` | `ErrorReport` | 顶栏告警 + 错误时间线 | 顶栏 + 抽屉 |
| `/fsm/log_event` | `FsmLogEvent` | 底部 console（最多 500 行） | 底栏 |
| `/vacuum/pressure` | `Float32MultiArray` | 侧栏左右真空压力曲线（最近 30s）| 侧栏卡片 |
| `/safety/status` | `SafetyStatus` | 顶栏：安全指示灯（estop/zone/collision/comm 四个小灯）| 顶栏徽章 |
| `/safety/estop` | `Bool` | 顶栏：急停红灯 | 顶栏徽章 |
| `/perception/health` | `PerceptionHealth` | 顶栏：视觉指示灯（camera/lidar/yolo/tf 四个小灯）；侧栏数值 | 顶栏徽章 + 侧栏 |

| `/vacuum/health` | `Bool` | 顶栏：吸盘指示灯（mock first 期常显示绿）| 顶栏徽章 |

## 3. 控制清单（写操作，受限）

UI 仅允许调以下 Service / Action goal：

| 控件 | 触发的接口 | 类型 | 权限 |
|---|---|---|---|
| ▶ 启动任务 | `/task/start` | `TaskControl.srv`（command="start"） | 任何人 |
| ⏸ 暂停 | `/task/pause` | `TaskControl.srv`（command="pause"）| 仅 AUTO_MODE |
| ⏵ 继续 | `/task/resume` | `TaskControl.srv`（command="resume"）| 仅 PAUSED |
| ⏹ 取消 | `/task/cancel` | `TaskControl.srv`（command="cancel"）| AUTO_MODE / PAUSED |
| 🔴 软急停 | `/safety/sw_estop` | `std_srvs/SetBool`（true=press，false=release）| 任何人 |
| 🔧 清错 | `/clear_error` | `ClearError.srv` | 仅 FAULT / E_STOP / EMERGENCY |
| 🧪 切 mock 模式（仅开发期）| `/mock_perception_adapter/set_mode` 等 | 自定义 srv | 调试模式 |

UI **不能**直接调：
- Nav2 的 `NavigateToPose` Action（必须经 strategy）
- `ExecutePairGrasp` Action（同上）
- `/chassis_node/*`、`/initialpose`、`/local_costmap/clear_*`（全部经 navigation_manager）
- `/vacuum/cmd`（除非 `command_source=SOURCE_MANUAL` 的开发模式按钮，且需 PAUSED）

### 3.1 软急停按钮

UI 的红色急停按钮按下时：

```text
1. UI 端立刻把按钮置成"已按"视觉态（不等 ack）
2. 调 safety_monitor_node 的 `/safety/sw_estop` Service（`std_srvs/SetBool`，`data=true` 表示 press，`data=false` 表示 release）
   safety_monitor 收到后切到 EMERGENCY，按 02 §10 行为发 /estop=true 与 /safety/estop=true
3. UI 等下一帧 /safety/estop==true → 按钮持续红
4. UI 端不直接发 /estop（避免与 safety_monitor 抢话题）
```

> 实现位置：`safety_monitor_node` expose `/safety/sw_estop` (`std_srvs/SetBool`)，UI 调它而不是直接发 `/estop`。这样保留"权威源唯一"原则。

### 3.2 启动任务的参数表单

```text
┌─ 启动任务 ─────────────────┐
│ task_id:    [auto / 可手填]│
│ start_wall: [0]            │
│ max_walls:  [0=不限制]     │
│ fixed_place_pose:          │
│   x [0.50]  y [0.00]  z [0.80]   │
│ params_json: [debug 自由扩展] │
│ □ dry_run                  │
│              [取消] [启动] │
└────────────────────────────┘
```

提交后调 `/task/start`，把 response 的 `accepted / message / current_task_state` 直接弹 toast。

### 4.1 顶栏徽章规格

| 徽章 | 来源 | 状态 | 颜色 |
|---|---|---|---|
| 系统模式 | `/fsm/system_state.current_state` | BOOTING / SELF_CHECK / STANDBY / AUTO_MODE / MANUAL_MODE / PAUSED / FAULT / E_STOP / SHUTDOWN | 见 §6 |
| 任务进度 | `/fsm/task_state.current_state` + wall_index/phase/elapsed | WAIT_TASK / RUN_TASK / ... | 灰/蓝 |
| 安全灯 | `/safety/status` | estop=false && zone_ok && comm_ok → 绿；任一 warn → 黄；estop=true → 红 | 绿/黄/红 |
| 视觉灯 | `/perception/health` | 全 ok → 绿；任一 false → 红；error_code=8020 (rate low) → 黄 | 绿/黄/红 |
| 导航灯 | `/fsm/nav_health` | True 绿 / False 红 | 绿/红 |
| 吸盘灯 | `/vacuum/health` | True 绿 / False 红 | 绿/红 |
| 软急停 | 按钮 | 未按下：白底红字 "急停"；按下后：红底白字 "已急停" | — |

### 4.2 主画布：5×5 网格

订 `/fsm/grid_snapshot` (`WallGridSnapshot`)。

```
渲染规则：
- 25 个 slot 排成 5 行 × 5 列，row=0 是顶部
- 每 slot 是一个 80×80 px 卡片，含：
    标题：row.col（如 "2.3"）
    状态徽章：UNKNOWN / OCCUPIED / REMOVED / FAILED / BLOCKED
    visible 灯（小圆点）：true 实心、false 空心
    confidence：进度条 0-1
    retry_count：右上角小数字
- 颜色（status）：
    UNKNOWN  → 浅灰
    OCCUPIED → 蓝
    REMOVED  → 绿
    FAILED   → 黄（带 retry_count）
    BLOCKED  → 红
- 当前 pair 的 left/right slot：加粗外边框 + 闪烁动画 1Hz
- 鼠标悬停：tooltip 显示 expected_pose_robot 与 latest_pose_robot 的 diff
```

收到 `/fsm/grasp_pair` 时，把 left_slot_id / right_slot_id 与当前 grid 对照，匹配的 slot 加高亮。

### 4.3 状态图侧栏（可折叠）

直接复用 06 Mermaid 图集。每个 FSM 一个 tab：

- RobotSystemFSM
- TaskFSM
- WallDestackingFSM（默认展开）
- WallMappingFSM / PhasePerceptionFSM / PairSelectionFSM / WallRecoveryFSM
- BaseNavigationFSM / PairGraspExecutionFSM
- SafetyMonitorFSM

实现：把 06 的 Mermaid 文本编译进前端 bundle（`mermaid-js` 库）；当前活跃状态用 CSS class 高亮。

> 不要在前端再画一遍状态图——直接用 06 文档的 Mermaid 字符串作为唯一来源，保证两份一致。

### 4.4 错误抽屉

订 `/fsm/error` (`ErrorReport`)，最近 50 条 FIFO。

```
┌─ 错误时间线 ─────────────────────────┐
│ 14:23:01  [ERROR ] 4010  ...        │
│           E_NAV_LOCALIZATION_LOST    │
│           strategy_node @ NAVIGATE_  │
│           recommended_recovery: RELOC│
│ 14:22:50  [WARN  ] 3221  ...        │
│           ...                        │
└──────────────────────────────────────┘
```

颜色 / 等级映射：

| level | 颜色 | 是否弹 toast |
|---|---|---|
| INFO  | 蓝 | 否 |
| WARN  | 黄 | 不弹（仅顶栏指示）|
| ERROR | 橙 | 弹 toast 5s |
| FATAL | 红 | 弹模态对话框，提示"调用 /clear_error" |
| ESTOP | 红闪 | 弹模态 + 顶栏所有灯红闪 |

### 4.5 底栏 console

订 `/fsm/log_event` (`FsmLogEvent`)。表格列：`ts | node | fsm | event | from→to | duration_ms | error_code | extra`。

- 默认显示最近 500 行；超出 FIFO 丢弃
- 每行点击展开 `extra_json`
- 顶部过滤器：node、event_type、最少 level
- 不做服务端搜索；纯前端筛

## 5. 子页与抽屉

### 5.1 任务历史（侧抽屉，从右滑出）

直接读节点 JSON 日志的 last N 行（通过 rosbridge 不容易做；改方案见下）。

**简化方案**：每个任务结束时，task_manager 在 `/fsm/log_event.event_type="task_summary"` 发一条总结，UI 累积保留最近 20 条。具体字段：

```json
{
  "task_id": "task_20260523_001",
  "result": "success",
  "walls_completed": 2,
  "total_boxes_picked": 50,
  "duration_sec": 540.0,
  "retry_total": 3,
  "fatal_count": 0
}
```

这样不必跨进程读文件，也不暴露日志路径给 UI。

### 5.2 调试面板（仅 mock 期）

通过 launch arg 决定是否开。打开后：

- 切 mock_perception 的 mode（OBSERVATION / LEFT_PHASE / RIGHT_PHASE / EMPTY / PARTIAL / JITTER）
- 调 mock_navigation_manager 的 inject_failure（下拉选 + duration 输入）
- 调 mock_pair_grasp 的 inject_failure
- 调 mock_vacuum_io 的 inject_failure
- 触发 mock_safety_button 的 press / release

实现：UI 检测到 `/mock_*/inject_failure` Service 在线即显示故障注入控件；`mock_safety_button` 通过 `/mock_safety_button/press` 与 `/mock_safety_button/release` 两个 Trigger Service 单独检测。线上模式（无 mock）自动隐藏。

### 5.3 网格 / pair 详情抽屉

点击网格上某 slot：右侧滑出抽屉显示该 slot 的全字段（来自 `/fsm/grid_snapshot.slots[i]`）。点击当前 pair 标题：抽屉显示完整 GraspPair。

## 7. 性能与连接策略

### 7.1 帧率限制

| Topic | UI 处理频率 | 策略 |
|---|---|---|
| `/fsm/system_state` / `/fsm/task_state` | 1Hz | 按需重渲染 |
| `/fsm/wall_destacking_state` / `/fsm/active_substate` | 5Hz | 节流 |
| `/fsm/grid_snapshot` | 状态变化时 | 直接渲染 |
| `/fsm/grasp_pair` | 每 pair 一次 | 直接渲染 |
| `/fsm/error` | 实时 | toast + 列表追加 |
| `/fsm/log_event` | 限 50/s | 超出丢弃旧的 |
| `/vacuum/pressure` | 20Hz → 抽样 5Hz | 最近 30s 滑窗 |
| `/perception/box_detections` | 5-10Hz → 抽样 2Hz | 仅调试面板用 |

> 限频在 UI 端做，不要让 publisher 改频率（会影响 strategy）。

### 7.2 重连

- rosbridge 断线时 UI 顶栏出现"已离线"横条
- 每 2s 重连一次；恢复后 UI 通过 TRANSIENT_LOCAL QoS 自动拿到最新状态快照
- 控制按钮在离线期禁用（灰底，hover 提示）

### 7.3 多窗口 / 多用户

允许同时多个 UI 实例连接同一 rosbridge：

- 状态广播都是 Topic，所有 UI 看到一致内容
- 写操作（`/task/*`、`/clear_error`、软急停）由 task_manager / safety_monitor 串行处理（service 调用本身互斥）
- UI 不做"谁先按谁拿到锁"的协调；交给后端 service 自己决定 accept / reject
- 每条写操作的 message 字段会回显到底栏 console，便于排查谁触发的

## 8. 安全与权限

### 8.1 写操作的最低门槛

本期不做用户登录 / RBAC。但 UI 必须做以下软约束：

| 操作 | 前置 | 实现 |
|---|---|---|
| `/task/start` | 当前 system_mode == STANDBY | 按钮在非 STANDBY 时禁用 |
| `/task/pause` | system_mode == AUTO_MODE | 同上 |
| `/task/resume` | system_mode == PAUSED | 同上 |
| `/task/cancel` | system_mode in {AUTO_MODE, PAUSED} | 同上 |
| `/clear_error` | system_mode in {FAULT, E_STOP} | 模态对话框确认 |
| 软急停 | 任何 | 不禁用，但二次确认（按住 1s 才生效，避免误触） |

后端最终判定权威；UI 的禁用只是 UX 层。

### 8.2 危险操作的 confirm

下列操作弹模态确认（"确认 / 取消"两按钮 + 倒计时 5s）：

- 软急停（按住 1s 触发，按错也好恢复，本项二次确认可选）
- `/clear_error`：列出失败原因 + 提示"将依次执行 5 阶段恢复"
- `/task/cancel`：列出"将取消 wall_X phase_Y 当前进度"
- 任何 `inject_failure` Service（仅调试模式）

### 8.3 不暴露的接口

UI 上**没有**任何按钮直接调以下接口（即便它们是 Service）：

- `/chassis_node/*`
- `/local_costmap/clear_*` / `/global_costmap/clear_*`
- `/initialpose`
- `/vacuum/cmd`
- `/lifecycle_manager_*/manage_nodes`

这些只能从代码内部走，避免人为破坏不变式。
