## 3. 总体策略

M2-preintegration 不直接改 FSM 主线接口，而是把真实后端藏在 adapter / manager 节点内部。

### 3.1 代码与工作区组织策略

后续机械臂规划、底盘导航和视觉感知的接入采用 **统一集成、边界隔离**：

```text
整机集成：
  使用当前 fsm_ws 作为 ROS2 colcon 集成工作区，fsm_config/launch 负责把 FSM、adapter、manager、mock / fake / real 后端组合起来。

代码归属：
  FSM 业务状态只写任务编排、策略决策和恢复逻辑。
  视觉、Nav2、MoveIt、底盘驱动、机械臂控制、真空硬件可以保留在独立包或外部 workspace。
  接入本系统时，通过 perception_adapter / navigation_manager / pair_grasp_execution / vacuum_io 转成 fsm_msgs 契约。

运行方式：
  开发期允许半真半 mock：一次只替换一个后端。
  真机期不改 strategy / task_manager，只调整 launch、参数和 backend_mode。
  若外部包太重或已有独立仓库，通过 ROS2 overlay 接入；不要为了"统一"把第三方栈复制进 FSM 业务包。
```

这条原则解决的是"完全独立工作空间"和"所有实现塞进一个空间"之间的取舍：

| 方案 | 结论 | 原因 |
|---|---|---|
| 各能力完全独立工作区，各自控制整机一部分 | 不采用 | 缺少统一任务语义、错误码和恢复链路，端到端问题难排查 |
| 所有算法、驱动、FSM 写进一个大 package / 大节点 | 不采用 | 故障域、开发归属、替换成本和测试粒度都会失控 |
| 统一 ROS graph / launch / fsm_msgs，能力实现按节点和包隔离 | 采用 | 可以统一验收整机行为，又能让视觉、导航、抓取独立演进和替换 |

### 3.2 backend_mode 策略

推荐给关键节点增加统一参数：

```yaml
backend_mode:
  mock        # M1 已有 mock 节点
  replay      # 从录制/构造数据发布真实消息形状
  fake_real   # 使用真实 client 逻辑，对接 fake server
  real        # 对接真实后端
```

当前阶段优先实现：

```text
perception_adapter_node: replay / real message path
navigation_manager_node: fake_real / real client path
pair_grasp_execution_node: dry_run / fake_real / real MoveIt skeleton
```

验收原则：

- 无真机时，必须能用 fake / replay 覆盖成功路径、失败路径、超时路径。
- 真机接入后，不改 FSM 上层接口，只把 `backend_mode` 切到 `real`。
- `colcon test` 和 M1 smoke 必须继续通过。

## 5. 不建议现在做的事

| 事项 | 原因 |
|---|---|
| 直接把旧代码拷进 FSM 业务状态机 | 会把 FSM 和硬件实现强耦合，后续很难替换 |
| 直接调用 `dual_arm_planner_node ~/plan_and_execute` | 该接口不接收目标，是 demo |
| 在策略层直接订 `/box_perception/result` | 会绕过 adapter，破坏 FSM 契约 |
| 在策略层直接读 Isaac USD prim 或 Isaac 私有 Topic | 会把仿真实现泄漏到业务状态机，后续真机/其他仿真后端无法替换 |
| 在 FSM 主线里直接发 `/cmd_vel` | 应由 Nav2 / navigation_manager / twist_mux 处理 |
| 现在规划真实吸盘 IO 后端 | 硬件未选型，接口容易反复变 |
