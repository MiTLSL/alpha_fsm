## 1. Frame 与单位约定

### 1.1 坐标系

> **FSM 契约层** 只用下表左列的"逻辑 frame 名"。物理 TF 树里的实际 frame 名由 fast_lio / lio_2d_mapper 决定（`camera_init`、`body`、`2d_body`），由 launch 层的静态 TF 桥接到逻辑名。FSM 业务代码 **不直接引用** `camera_init` / `body` / `2d_body`。

| 逻辑 frame_id（FSM 用） | 物理 TF 上游来源 | 桥接方式 | 用途 |
|---|---|---|---|
| `map` | `nav2_amcl/amcl` 发 `map → camera_init` | `navigation_manager_node` 对外转换为逻辑 `map` | 全局参考系 |
| `odom` | `fast_lio` 发 `camera_init → body` | 不要求物理 TF 树存在 `odom`；仅在 FSM 消息字段中作为逻辑短时系使用 | 短时漂移系 |
| `base_link` | fast_lio 的 `body` | launch 静态 TF：`body → base_link`（identity 或仅平移） | 机器人本体（底盘中心） |
| `2d_base_link` | `lio_2d_mapper/lidar_tf_node` 发 `map → 2d_body` | launch 静态 TF：`2d_body → 2d_base_link` | Nav2 的 `robot_base_frame`，xy + yaw，Z=0 |
| `camera_link` | URDF 静态 TF | 直接 | 相机光心 |
| `livox_frame` | `livox_ros_driver2` | 直接 | 雷达光心 |
| `left_tcp` / `right_tcp` | URDF 静态 TF | 直接 | 左右臂末端（对应 `left_v5_tool0` / `right_v5_tool0`） |
| `wall_<index>_frame` | 动态 | WallMappingFSM 估计后 broadcast | 当前墙面坐标系，原点 = 墙中心，X 朝向相机，Y 朝左，Z 朝上 |
| `container_frame` | 静态（启动时标定） | tf | 集装箱坐标系（可选） |

> **强约束**：
> - launch 层（`fsm_config/launch/bringup_full.launch.py`）必须显式发布静态 TF `body → base_link`、`2d_body → 2d_base_link`。**不要**额外静态发布 `map → odom` 或 `camera_init → odom` 去覆盖 AMCL / Fast-LIO 的动态链路。
> - Nav2 的 `robot_base_frame` 在 `nav2_params.yaml` 里写的是 `2d_body`；`navigation_manager_node` 在和 Nav2 交互时使用 `2d_base_link` / `2d_body`，对外（向策略层）一律用 `base_link`。
> - 任何 FSM 业务字段（如 `*_pose_robot`）的 `header.frame_id` 写 `"base_link"`，**绝不写 `"body"` 或 `"2d_body"`**。

### 1.2 字段命名规则

```text
所有 pose 字段必须带 frame 后缀：
   left_box_pose_robot      表示在 base_link 系下
   left_box_pose_world      表示在 map 系下
   wall_pose_camera         表示在 camera_link 系下

对应 frame_id：
   _robot   → "base_link"
   _world   → "map"
   _camera  → "camera_link"
   _wall    → "wall_<index>_frame"
```

### 1.3 单位

```text
长度       米 (m)
角度       弧度 (rad)
线速度     m/s
角速度     rad/s
时间戳     builtin_interfaces/Time（ROS2 标准）
持续时间   builtin_interfaces/Duration
压力       kPa（vacuum 用）
置信度     [0.0, 1.0] float32
```

### 1.4 Pose 表示

跨节点统一用 `geometry_msgs/PoseStamped`：

```text
header.frame_id       明确写 frame，禁止留空
header.stamp          数据时间戳
pose.position         x, y, z (m)
pose.orientation      x, y, z, w（四元数，归一化）
```

不允许用欧拉角传输（除非是 nav_goal 里的 yaw 调整量，明确单字段）。

## 2. 自定义消息定义

### 2.1 msg

#### 2.1.1 BoxDetection.msg

来自 **`perception_adapter_node`**（不是直接来自 box_perception 进程）的单个箱子检测结果。

```text
# BoxDetection.msg
std_msgs/Header header                    # frame_id 必须是 "base_link"（adapter 已用 TF2 转过）
string detection_id                       # 该帧内唯一 id（如 "frame_1234_box_03"）
geometry_msgs/PoseStamped pose            # 6D pose, frame_id="base_link"
geometry_msgs/Vector3 size                # 长宽高 (m)，标准箱约 0.4×0.4×0.4
float32 confidence                        # YOLO 置信度 [0.0, 1.0]
string class_label                        # "box" / 其他
bool   pose_valid                         # false 表示 pose 估计失败但 bbox 仍可用
```

> **来源说明**：上游 `box_perception/perception_node` 实际发布的是 `box_perception_msgs/BoxPerceptionResult`（频率 ≈ 点云帧率，frame_id=`body`，且不含 box `size`、不发 `BoxDetectionArray` 形状）。`perception_adapter_node` 负责：
> 1. 订 `/box_perception/result`（上游真实 Topic）
> 2. 用 TF2 把 `body` 系的 `nearest_face_center` / `nearest_face_normal` 转到 `base_link`
> 3. 由箱面中心 + 法向 + 标定 box 尺寸（`config/business.yaml` 的 `business.box_size`）反推箱体中心 6D pose
> 4. 重新打包成 `fsm_msgs/BoxDetectionArray` 发到 `/perception/box_detections`
> 5. 兜底：上游 BoxResult 的 `face_inlier_count_*` 过低 / `pose_valid=false` 时仍发出，但置 `pose_valid=false`

#### 2.1.2 BoxDetectionArray.msg

```text
# BoxDetectionArray.msg
std_msgs/Header header                    # 整批检测的时间戳；frame_id="base_link"
fsm_msgs/BoxDetection[] detections
uint32  frame_seq                         # 帧序号，单调递增，便于策略层判帧间隔
float32 inference_latency_ms              # 该帧 YOLO + 点云提取 + TF 转换的总耗时（仅供 debug）
```

> 注：`perception_adapter_node` 以恒定频率（建议 5-10Hz，取决于上游 `/box_perception/result` 的速率）持续 publish 本 Topic，**不提供 trigger 类 Service**。
> 策略层订阅后自行开窗收 N 帧做融合判定（详见 WallMappingFSM / PhasePerceptionFSM）。
> 当前底盘是远观察位还是近抓取位，由策略层自己掌握，adapter 不感知作业阶段。

#### 2.1.13 PerceptionHealth.msg

由 **`perception_adapter_node`** 自检上游话题后发布（上游 `box_perception/perception_node` **本身不发任何 health 心跳**，必须由 adapter 兜底）。

```text
# PerceptionHealth.msg
std_msgs/Header header
bool    camera_ok                         # /camera/camera/color/image_raw 在 ≤500ms 内有新帧
bool    lidar_ok                          # /cloud_registered_body 在 ≤500ms 内有新帧
bool    yolo_ok                           # /box_perception/result 在 ≤1s 内有新帧，且非 fallback stub
bool    tf_ok                             # TF body→base_link 与 camera_link→base_link 可解析、stamp 新鲜
float32 detection_publish_rate_hz         # adapter 自己输出的 /perception/box_detections 实际发布频率
float32 camera_frame_age_ms               # 最近一帧相机数据距今多少 ms
float32 lidar_frame_age_ms                # 最近一帧雷达数据距今多少 ms
float32 upstream_result_age_ms            # 最近一帧 /box_perception/result 距今多少 ms
uint16  error_code                        # 0 = 健康；非 0 见 04 错误码 8xxx 区
string  details_json
```

**adapter 内的判定阈值**（配置项放 `fsm_config/business.yaml`，下面是默认值）：

| 字段 | 健康阈值 | error_code |
|---|---|---|
| `camera_ok` | image 帧间隔 < 500ms | `E_EXT_PERC_CAMERA_FAIL` |
| `lidar_ok` | cloud_registered_body 帧间隔 < 500ms | `E_EXT_PERC_LIDAR_FAIL` |
| `yolo_ok` | result 帧间隔 < 1000ms（且非 stub） | `E_EXT_PERC_YOLO_FAIL` |
| `tf_ok` | TF 查询成功且 stamp 在 200ms 内 | `E_COMM_TF_LOOKUP_FAIL` |

**频率**：1-2Hz 心跳，**任一布尔字段翻转时立刻额外发一次**。
**QoS**：`RELIABLE + TRANSIENT_LOCAL`（晚启动的订阅者也能拿到最新状态）。

#### 2.2.4 ClearError.srv

```text
# ClearError.srv
uint16  error_code                        # 0 表示清除全部
string  reason
---
bool    cleared
string  message
uint8   stage_reached                     # 见下方常量；失败时停在哪一阶段
```

**`stage_reached` 常量**（FSM `RobotSystemFSM` `clear_error` 顺序协议固定，见 02 §1.3.x）：

```text
uint8 STAGE_NONE             = 0   # 还没开始
uint8 STAGE_ESTOP_RELEASED   = 1   # 已发 /estop=False，twist_mux /stop=False 已观察到
uint8 STAGE_ACTIONS_CANCELED = 2   # 所有 outstanding Action 已 cancel 完成
uint8 STAGE_FAULT_RESET      = 3   # /chassis_node/reset_fault 成功，fault_a/b 均为 False
uint8 STAGE_CHASSIS_ENABLED  = 4   # /chassis_node/enable 成功，status.enabled=True
uint8 STAGE_SELF_CHECK       = 5   # 已跳到 SELF_CHECK，本次 clear_error 完成
```

> `task_manager_node` 是 `/clear_error` 的唯一 Service owner，必须按 `1 → 2 → 3 → 4 → 5` 顺序推进；任一步失败立即返回 `cleared=false` 并填 `stage_reached`。阶段 1/3/4 通过 §2.2.5 的内部服务委托给 `navigation_manager_node` 执行。详细超时与降级见 02 §1.3.x。

> **FINE_ALIGN 子状态对外行为**：进入前 `navigation_manager_node` 必须确保 Nav2 coarse goal 已经成功返回，或在达到粗定位容差后内部 cancel Nav2 goal 并等待 cancel ack；进入后只通过 `/cmd_vel_align` 发小速度。若 FINE_ALIGN 失败，需要重新发一个新的 Nav2 goal 或直接返回失败，**不依赖旧 Nav2 goal 继续收尾**。详细见 02 §8 BaseNavigationFSM 内部状态。

## 6. 字段约束与验证规则

下面这些是 fsm_msgs 层不强制（消息体本身允许），但业务层必须遵守的不变式：

### 6.1 GraspPair

```text
1. left_box_pose_robot.position.y > right_box_pose_robot.position.y
   （左手抓的箱子 Y 比右手大；除非单臂模式）
2. 单臂模式 (grasp_mode != DUAL) 时，未使用的 slot_id 为空字符串、pose 为全 0
3. wall_index 与左右两个 slot_id 的前缀必须一致
4. phase 字段表示执行该 pair 时底盘所在作业位；active slot 的 col_index 必须落在该 phase 配置列内（LEFT_PHASE → col ∈ left_phase_cols，RIGHT_PHASE → col ∈ right_phase_cols）
5. 策略层生成 pair 前必须保证抓取后 5 列剩余高度的相邻列差值 <= `business.max_adjacent_column_height_delta`；双抓 pair 还必须保证任意单边成功后的高度也满足该约束
```

### 6.2 WallGridSnapshot

```text
1. slots 长度必须 = rows * cols = 25
2. slots 按 row-major 排列：index = row * cols + col
3. wall_frame_pose 与每个 slot 的 expected_pose_robot 必须基于同一 frame（base_link）
```

### 6.3 NavigateToPose Action

```text
1. goal_type ∈ {OBSERVATION, LEFT_PHASE, RIGHT_PHASE, NEXT_WALL, SAFE, RECOVERY}
2. goal_type 为 LEFT/RIGHT_PHASE 时，phase 字段必填，且 wall_frame_pose 必填
3. target_pose 的 frame_id 必须 = "map"
4. 如果 require_fine_alignment=true，BaseNavigationFSM 必须执行 WallFineAlignment 子流程
```

### 6.4 ExecutePairGrasp Action

```text
1. grasp_pair.pair_id 必须非空
2. 两个 box pose 的 frame_id 必须 = "base_link"
3. fixed_place_pose_robot 在策略层由 ctx.config.fixed_place_pose 注入，frame_id="base_link"
4. dry_run=true 时只规划不执行，但仍返回完整 result
```

### 6.5 PairGraspResult

```text
1. result_code = SUCCESS_BOTH ⇒ left_result=SUCCESS AND right_result=SUCCESS
2. result_code = SUCCESS_LEFT_ONLY ⇒ left_result=SUCCESS AND right_result=FAIL
3. result_code = ESTOP ⇒ 立即上报，不再执行后续阶段
4. result_code = CANCELLED ⇒ 用户取消或上层 cancel；不得复用 ESTOP
5. failed_stage 在 result_code 不属于 SUCCESS_* 且不是 CANCELLED 时必须非空
```

## 10. twist_mux.yaml 扩展（FSM 落地前需要做的一次性改动）

> 当前 `ros2_ws/src/alfa_robot_twist_mux/config/twist_mux.yaml` 只有 teleop / nav / stop 三路。要支持 FINE_ALIGN，需要新增一个 `align` 通道。**这是 FSM 启动前必须落地的部署改动**，但不要在 FSM 业务代码里做——交给部署/A 工程师在 twist_mux 包内改 yaml。

建议改后内容：

```yaml
twist_mux:
  ros__parameters:
    topics:
      teleop:
        topic: /cmd_vel_teleop
        timeout: 0.5
        priority: 100
      align:                       # 新增
        topic: /cmd_vel_align
        timeout: 0.5
        priority: 50
      nav:
        topic: /cmd_vel_nav
        timeout: 0.5
        priority: 10
      stop:
        topic: /cmd_vel_estop
        timeout: 0.1
        priority: 255
    locks:
      estop_lock:
        topic: /stop
        timeout: 0.0
        priority: 255
```

> 优先级语义：`align=50` 高于 `nav=10`，FINE_ALIGN 进行时即使 Nav2 还在发 `/cmd_vel_nav` 也会被压住；teleop=100 高于 align，遥控仍可强制接管；estop=255 高于一切。
>
> 改完 yaml 后**不需要重建 twist_mux 源码**，重启该节点即可生效。
