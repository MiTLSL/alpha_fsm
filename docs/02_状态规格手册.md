# 02 状态规格手册

> 本文档为每个状态提供"规格卡"，作为代码实现的契约。每张卡有 13 个字段。你负责的 FSM 详写；工程师 A/B 负责的 FSM 仅给精简卡（保留接口与跳转，内部细节由其参考既有代码补）。

---

## 0. 规格卡格式

每个状态严格按以下格式写：

```text
### <状态名>
- 所属 FSM    ：<父/子 FSM 名>
- 进入条件    ：<什么情况下能进入这个状态>
- 退出条件    ：<什么情况下离开>
- 成功跳转    ：<状态名 / 多个时按 transition 列出>
- 失败跳转    ：<状态名>
- 超时        ：<秒>，超时跳转：<状态名>
- 重试上限    ：<次>
- 可能异常码  ：[error_code 列表]
- ROS2 接口   ：<本状态调用/订阅/发布的接口>
- Context 读  ：<从 ctx 读哪些字段>
- Context 写  ：<向 ctx 写哪些字段>
- 日志要点    ：<除标准 enter/exit 外，本状态需要额外记录的 metric>
- 备注        ：<关键实现注意点>
```

## 1. RobotSystemFSM（你负责）

> 所在节点：`task_manager_node`。负责整机生命周期与模式。

### 1.1 状态总览

```text
BOOTING → SELF_CHECK → STANDBY → AUTO_MODE
                                  ↕
                                MANUAL_MODE
                                  ↓
                                PAUSED
                                  ↓
                                FAULT
                                  ↓
                                E_STOP
                                  ↓
                                SHUTDOWN
```

### 1.2 详细状态卡

#### BOOTING
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：节点启动时初始状态
- 退出条件    ：所有底层节点心跳齐 → SELF_CHECK
- 成功跳转    ：SELF_CHECK
- 失败跳转    ：FAULT
- 超时        ：30s，超时跳转：FAULT
- 重试上限    ：0
- 可能异常码  ：[E_SYS_BOOT_FAIL, E_SYS_HARDWARE_NOT_READY, E_COMM_NODE_OFFLINE]
- ROS2 接口   ：sub `/safety/status`、sub `/perception/health`、查询其他节点心跳
- Context 读  ：—
- Context 写  ：`system_mode = "BOOTING"`，`task_state = "WAIT_TASK"`
- 日志要点    ：每个节点上线/超时
- 备注        ：等所有外部节点 ready 后再进 SELF_CHECK，避免后续状态反复检查

#### SELF_CHECK
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：BOOTING 完成
- 退出条件    ：自检全部通过 + 底盘已使能 → STANDBY；任一项失败 → FAULT
- 成功跳转    ：STANDBY
- 失败跳转    ：FAULT
- 超时        ：20s，超时跳转：FAULT
- 重试上限    ：0
- 可能异常码  ：[E_SYS_BOOT_FAIL, E_SYS_CONFIG_INVALID, E_EXT_PERC_OFFLINE, E_EXT_PERC_CAMERA_FAIL, E_EXT_PERC_LIDAR_FAIL, E_NAV_LIFECYCLE_NOT_ACTIVE, E_CHASSIS_ENABLE_FAIL]
- ROS2 接口   ：
  - 读 `/perception/health`、`/safety/status`、tf 是否齐全
  - 调 `/lifecycle_manager_navigation/is_active` 与 `/lifecycle_manager_localization/is_active`（两个都必须 active）
  - 调 `/chassis_node/enable`（Trigger），订 `/chassis_node/status` 确认 `enabled=True`
- Context 读  ：safety_status, perception_health
- Context 写  ：system_mode = "SELF_CHECK"
- 日志要点    ：自检每一项通过/失败 + chassis enable 结果
- 备注        ：
  - 本状态是"门禁"，宁严勿松
  - 自检顺序固定：① perception health → ② tf 齐全 → ③ Nav2 两个 lifecycle manager 都 active → ④ 调 `/chassis_node/enable` → ⑤ 等 `/chassis_node/status.enabled=True`（≤2s）→ 通过
  - 第④步**必须做**：底盘默认 `auto_enable_on_start=false`，不调 enable 整套 nav 链上线但底盘不动
  - clear_error 流程进入 SELF_CHECK 时也走同一流程（所以 SELF_CHECK 实现要幂等）

#### STANDBY
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：自检通过
- 退出条件    ：收到 `/task/start` Service → AUTO_MODE；收到手动模式切换请求 → MANUAL_MODE
- 成功跳转    ：AUTO_MODE / MANUAL_MODE
- 失败跳转    ：—
- 超时        ：无（长期等待状态）
- 重试上限    ：0
- 可能异常码  ：[]
- ROS2 接口   ：sub `/safety/status`、`/task/start` Service handler
- Context 读  ：—
- Context 写  ：system_mode = "STANDBY"
- 日志要点    ：进入 STANDBY 心跳 1Hz
- 备注        ：唯一允许接受新任务的状态

#### AUTO_MODE
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：用户启动任务
- 退出条件    ：TaskFSM 完成 → STANDBY；FATAL 错误 → FAULT；急停 → E_STOP；用户暂停 → PAUSED
- 成功跳转    ：STANDBY（任务完成）
- 失败跳转    ：FAULT
- 超时        ：无
- 重试上限    ：0
- 可能异常码  ：[E_TASK_CHILD_FAILED, E_TASK_CHILD_TIMEOUT]
- ROS2 接口   ：内部调用 TaskFSM（同进程）
- Context 读  ：safety_status
- Context 写  ：system_mode = "AUTO"
- 日志要点    ：TaskFSM 进度
- 备注        ：本状态实际工作交给 TaskFSM；监听急停信号优先级最高

#### MANUAL_MODE
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：用户切换到手动模式
- 退出条件    ：用户切回自动模式 → STANDBY
- 成功跳转    ：STANDBY
- 失败跳转    ：FAULT
- 超时        ：无
- 重试上限    ：0
- 可能异常码  ：[E_MAN_OVERRIDE]
- ROS2 接口   ：—
- Context 读  ：—
- Context 写  ：system_mode = "MANUAL"
- 日志要点    ：进入/退出
- 备注        ：本期 Demo 可仅占位实现，不做完整 manual 流程

#### PAUSED
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：AUTO_MODE 收到 `/task/pause`
- 退出条件    ：收到 `/task/resume` → AUTO_MODE；超时 → FAULT
- 成功跳转    ：AUTO_MODE
- 失败跳转    ：FAULT
- 超时        ：300s（参数化），超时跳转：FAULT
- 重试上限    ：0
- 可能异常码  ：[E_TASK_PAUSE_TIMEOUT]
- ROS2 接口   ：—
- Context 读  ：pause_requested
- Context 写  ：system_mode = "PAUSED"
- 日志要点    ：进入/退出 + 暂停时长
- 备注        ：暂停时所有正在执行的 Action 必须 cancel；恢复时 TaskFSM 决定如何续

#### FAULT
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：FATAL 错误 / 自检失败 / 暂停超时
- 退出条件    ：人工 `/clear_error` Service 调用、五阶段全部成功 → SELF_CHECK
- 成功跳转    ：SELF_CHECK
- 失败跳转    ：—（停在 FAULT，等待人工再触发或硬件介入）
- 超时        ：无
- 重试上限    ：0
- 可能异常码  ：携带 last_error 上报
- ROS2 接口   ：`/clear_error` Service handler；按 §1.3.x 顺序调用 navigation_manager 的 `/nav/base_recovery` 内部服务
- Context 读  ：last_error
- Context 写  ：system_mode = "FAULT"；clear_error 各阶段 stage_reached 写入 ctx
- 日志要点    ：进入时记录 last_error 完整快照；clear_error 每阶段写一条
- 备注        ：必须显式人工清错；clear_error 协议见 §1.3.x，**严禁简化为一步切状态**

#### E_STOP
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：硬件/软件急停
- 退出条件    ：急停解除（`/safety/estop=false`）且 `/clear_error` 五阶段全成 → SELF_CHECK
- 成功跳转    ：SELF_CHECK
- 失败跳转    ：—
- 超时        ：无
- 重试上限    ：0
- 可能异常码  ：[E_SAFETY_ESTOP_HW, E_SAFETY_ESTOP_SW]
- ROS2 接口   ：`/clear_error` Service handler；按 §1.3.x 顺序调用 `/nav/base_recovery`
- Context 读  ：—
- Context 写  ：system_mode = "E_STOP"
- 日志要点    ：急停源、当时各 FSM 状态快照
- 备注        ：
  - 进入时必须广播 abort 给所有子 FSM
  - **解除时必须严格按 §1.3.x 五阶段**：① 发 /estop=False → ② cancel 所有 Action → ③ /chassis_node/reset_fault → ④ /chassis_node/enable → ⑤ 重做 SELF_CHECK
  - 缺第①步：twist_mux 锁不释放，底盘永远收不到非零速
  - 缺第③/④步：底盘 fault 位仍 True 或未 enable，Nav2 规划成功但底盘不动

### 1.3.x clear_error 五阶段协议（FAULT / E_STOP 共用）

`/clear_error` Service 的实现**必须**严格按以下顺序，任一阶段失败立即返回 `cleared=false` 并填写 `stage_reached`，FSM 停在当前父状态等待下一次 `/clear_error`：

| 阶段 | 编号（=`stage_reached`）| 动作 | 等待条件 | 超时 | 失败时返回的 error_code |
|---:|---:|---|---|---:|---|
| 1 | `STAGE_ESTOP_RELEASED=1` | 发 `/estop=False` | 订阅 `/stop` 看到 `False`（twist_mux 锁释放） | 1s | `E_SAFETY_ESTOP_LOCK_STUCK` |
| 2 | `STAGE_ACTIONS_CANCELED=2` | 广播 abort 给所有子 FSM；cancel `/navigate_to_pose` 与 `/execute_pair_grasp` 的 outstanding goal | 收到所有 cancel ack（或 goal 早已无效） | 2s | `E_SYS_ACTION_CANCEL_TIMEOUT` |
| 3 | `STAGE_FAULT_RESET=3` | 调 `/chassis_node/reset_fault` Trigger | `/chassis_node/status.fault_a=False && fault_b=False` | 2s | `E_CHASSIS_FAULT_RESET_FAIL` |
| 4 | `STAGE_CHASSIS_ENABLED=4` | 调 `/chassis_node/enable` Trigger | `/chassis_node/status.enabled=True` | 2s | `E_CHASSIS_ENABLE_FAIL` |
| 5 | `STAGE_SELF_CHECK=5` | 跳转 SELF_CHECK 状态（不是直接回 STANDBY） | 进入 SELF_CHECK 完成 enter | 1s | `E_SYS_SELF_CHECK_REENTRY_FAIL` |

**实现位置**：`task_manager_node` 是 `/clear_error` 的唯一 Service owner。阶段 1、3、4 的实际 ROS 调用由 `navigation_manager_node` 的 `/nav/base_recovery` 内部服务提供（它持有 estop / chassis 句柄）；阶段 2 由 `task_manager_node` 自己广播给 `wall_destacking_strategy_node` 与 `pair_grasp_execution_node`；阶段 5 是 RobotSystemFSM 本地状态跳转。

**幂等性**：每一阶段都要幂等。比如重复发 `/estop=False` 不应使任何状态变坏；重复调 `enable` 应当返回 `success=true`（chassis_node 已经 enabled 时返回 true 即可）。

**日志**：每个阶段的"开始 / 等待 / 完成"都各写一条 `FsmLogEvent`，`event_type="metric"`，`extra_json={"clear_error_stage": <编号>}`。便于复盘"卡在哪一步"。

#### SHUTDOWN
- 所属 FSM    ：RobotSystemFSM
- 进入条件    ：用户关机命令
- 退出条件    ：清理完毕进程退出
- 成功跳转    ：（终态）
- 失败跳转    ：—
- 超时        ：5s
- 重试上限    ：0
- 可能异常码  ：[]
- ROS2 接口   ：—
- Context 读  ：—
- Context 写  ：system_mode = "SHUTDOWN"
- 日志要点    ：关机前最终状态快照
- 备注        ：本期 Demo 可省略，靠 Ctrl+C

## 2. TaskFSM（你负责）

> 所在节点：`task_manager_node`。负责一个拆垛任务的生命周期。

### 2.1 状态总览

```text
WAIT_TASK → ACCEPT_TASK → VALIDATE_TASK → PREPARE_TASK
                                              ↓
COMPLETE_TASK ← VERIFY_TASK_RESULT ← RUN_TASK
     ↓                                   ↓
WAIT_TASK                              FAIL_TASK or CANCEL_TASK
                                          ↓
                                       WAIT_TASK
```

### 2.2 详细状态卡

#### WAIT_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：FSM 启动初始 / 上一任务结束
- 退出条件    ：收到 `/task/start` → ACCEPT_TASK
- 成功跳转    ：ACCEPT_TASK
- 失败跳转    ：—
- 超时        ：无
- 可能异常码  ：[]
- ROS2 接口   ：sub `/task/start`
- Context 读  ：—
- Context 写  ：task_state = "WAIT_TASK"
- 日志要点    ：进入 + 心跳 1Hz
- 备注        ：唯一允许接收新任务的入口

#### ACCEPT_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：收到任务请求
- 退出条件    ：参数解析成功 → VALIDATE_TASK
- 成功跳转    ：VALIDATE_TASK
- 失败跳转    ：FAIL_TASK（参数解析错）
- 超时        ：1s
- 可能异常码  ：[E_TASK_VALIDATE_FAIL]
- ROS2 接口   ：—
- Context 读  ：—
- Context 写  ：task_id, task_start_time, fixed_place_pose
- 日志要点    ：task_id、入参 JSON
- 备注        ：本状态只解析，不做业务校验

#### VALIDATE_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：参数已解析
- 退出条件    ：所有前置条件通过 → PREPARE_TASK
- 成功跳转    ：PREPARE_TASK
- 失败跳转    ：FAIL_TASK
- 超时        ：5s
- 可能异常码  ：[E_TASK_VALIDATE_FAIL, E_TASK_PRECONDITION_FAIL]
- ROS2 接口   ：查询 `/safety/status`、`/perception/health`
- Context 读  ：safety_status
- Context 写  ：—
- 日志要点    ：每个前置条件结果
- 备注        ：检查 system_mode==AUTO、安全 OK、perception OK、tf 齐全

#### PREPARE_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：校验通过
- 退出条件    ：准备完毕 → RUN_TASK
- 成功跳转    ：RUN_TASK
- 失败跳转    ：FAIL_TASK
- 超时        ：3s
- 可能异常码  ：[E_TASK_PRECONDITION_FAIL]
- ROS2 接口   ：—
- Context 读  ：task_id
- Context 写  ：初始化 task 级日志 / 计数器
- 日志要点    ：task_start
- 备注        ：本期可极简，仅初始化日志文件、清空 wall_index

#### RUN_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：准备完毕
- 退出条件    ：WallDestackingFSM Action 返回 → VERIFY_TASK_RESULT
- 成功跳转    ：VERIFY_TASK_RESULT
- 失败跳转    ：FAIL_TASK
- 超时        ：1800s（参数化）
- 可能异常码  ：[E_TASK_CHILD_FAILED, E_TASK_CHILD_TIMEOUT]
- ROS2 接口   ：Action client `/run_wall_destacking`
- Context 读  ：task_id, fixed_place_pose, pause_requested, cancel_requested
- Context 写  ：wall_destacking_goal_handle, wall_destacking_result
- 日志要点    ：Action feedback 中 wall_index/phase 变化、metric: 总耗时
- 备注        ：暂停/取消请求 → cancel goal；急停 → cancel goal + 直接跳 FAIL_TASK

#### VERIFY_TASK_RESULT
- 所属 FSM    ：TaskFSM
- 进入条件    ：子 Action 完成
- 退出条件    ：结果合法 → COMPLETE_TASK；失败 → FAIL_TASK
- 成功跳转    ：COMPLETE_TASK
- 失败跳转    ：FAIL_TASK
- 超时        ：2s
- 可能异常码  ：[E_TASK_CHILD_FAILED]
- ROS2 接口   ：—
- Context 读  ：wall_destacking_result
- Context 写  ：—
- 日志要点    ：result.walls_completed, total_boxes_picked
- 备注        ：仅做合法性检查，不做重试

#### COMPLETE_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：结果合法
- 退出条件    ：上报完毕 → WAIT_TASK
- 成功跳转    ：WAIT_TASK
- 失败跳转    ：—
- 超时        ：2s
- 可能异常码  ：[]
- ROS2 接口   ：通过 `/task/start` Service response 返回（如还在 hold）
- Context 读  ：task_id, task_start_time
- Context 写  ：—
- 日志要点    ：task_summary：总耗时、抓取数、失败数、retry 数
- 备注        ：写一份 task summary 到日志根目录

#### FAIL_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：任何失败路径
- 退出条件    ：上报完毕 → WAIT_TASK 或 RobotSystemFSM.FAULT
- 成功跳转    ：WAIT_TASK
- 失败跳转    ：—
- 超时        ：2s
- 可能异常码  ：携带 last_error
- ROS2 接口   ：pub `/fsm/error`
- Context 读  ：last_error
- Context 写  ：—
- 日志要点    ：完整故障快照
- 备注        ：FATAL 上报到 RobotSystemFSM 触发 FAULT

#### CANCEL_TASK
- 所属 FSM    ：TaskFSM
- 进入条件    ：收到 `/task/cancel`
- 退出条件    ：cancel 完毕 → WAIT_TASK
- 成功跳转    ：WAIT_TASK
- 失败跳转    ：FAIL_TASK
- 超时        ：10s
- 可能异常码  ：[E_MAN_CANCELLED]
- ROS2 接口   ：cancel `/run_wall_destacking` goal
- Context 读  ：wall_destacking_goal_handle
- Context 写  ：—
- 日志要点    ：cancel 原因
- 备注        ：必须等 Action server 真正 cancel 完成再退

## 3. WallDestackingFSM（你负责，核心）

> 所在节点：`wall_destacking_strategy_node`。整面货箱墙的策略编排，是本系统最重要的 FSM。

### 3.1 状态总览

```text
INIT_WALL_TASK
  → NAVIGATE_TO_OBSERVATION_POSE
  → RUN_WALL_MAPPING
  → CHECK_WALL_VALID
  → NAVIGATE_TO_PHASE_WORKPOSE
  → RUN_PHASE_PERCEPTION
  → RUN_PAIR_SELECTION
      ├→ 推荐 phase 与当前作业位不同 DECIDE_NEXT_PHASE → NAVIGATE_TO_PHASE_WORKPOSE
      └→ 推荐 phase 与当前作业位相同 DISPATCH_PAIR_GRASP
  → WAIT_PAIR_GRASP_RESULT
  → UPDATE_GRID_AFTER_GRASP
  → DECIDE_NEXT_PAIR
      ├→ wall 仍有可抓箱 RUN_PHASE_PERCEPTION
      └→ wall 已完成 DECIDE_NEXT_WALL
                         ├→ 下一面 NAVIGATE_TO_OBSERVATION_POSE
                         └→ VERIFY_TASK_COMPLETE
                                ├→ WALL_DONE
                                └→ RUN_WALL_MAPPING（重确认）

异常出口：任何状态 FAILURE → WALL_ERROR_HANDLE → 由 WallRecoveryFSM 处理
```

策略原则：
- phase 表示底盘作业位偏好，不再表示必须先清空的一侧。
- PairSelection 每轮分别为左右作业位生成推荐候选；作业位内按“上层优先、左侧优先”排序。
- 全局 5 列高度安全是硬约束：候选抓取后任意相邻列剩余高度差不得超过 `business.max_adjacent_column_height_delta`，默认 1。
- 若左右作业位都有安全候选，优先当前底盘所在作业位；若当前作业位不安全而另一侧安全，则切换作业位。
- 双抓不满足高度安全时允许降级为单抓；当前 phase 只剩一个顶部箱时允许单抓（受 `allow_single_arm_grasp` 控制）。

### 3.2 详细状态卡

#### INIT_WALL_TASK
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：Action goal 接收完毕
- 退出条件    ：ctx 初始化完毕 → NAVIGATE_TO_OBSERVATION_POSE
- 成功跳转    ：NAVIGATE_TO_OBSERVATION_POSE
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：1s
- 可能异常码  ：[E_WALL_STATE_TIMEOUT]
- ROS2 接口   ：—
- Context 读  ：Action goal 字段
- Context 写  ：task_id, wall_index=0, current_phase=LEFT_PHASE, empty_confirm_count=0
- 日志要点    ：task_id 与 goal 摘要
- 备注        ：所有 ctx 字段恢复初始值，防止跨任务污染

#### NAVIGATE_TO_OBSERVATION_POSE
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：开始新 wall / 任务完成确认重扫
- 退出条件    ：导航 Action 成功 → RUN_WALL_MAPPING
- 成功跳转    ：RUN_WALL_MAPPING
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：60s（参数化，含路径+对齐）
- 重试上限    ：2（由 ErrorHandler 控制）
- 可能异常码  ：[E_NAV_GOAL_REJECTED, E_NAV_GOAL_TIMEOUT, E_NAV_PATH_PLAN_FAIL, E_NAV_LOCALIZATION_LOST, E_NAV_WORKPOSE_INVALID]
- ROS2 接口   ：Action client `/navigate_to_pose`，goal_type=OBSERVATION
- Context 读  ：wall_index, config.observation_pose
- Context 写  ：last_nav_result
- 日志要点    ：发出 goal、收到 feedback、最终 actual_pose
- 备注        ：goal_type=OBSERVATION 不需要 fine_alignment

#### RUN_WALL_MAPPING
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：观察位到达
- 退出条件    ：子 FSM 返回 → CHECK_WALL_VALID
- 成功跳转    ：CHECK_WALL_VALID
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：30s（子 FSM 自带超时，本层只兜底）
- 重试上限    ：1（外层 ErrorHandler）
- 可能异常码  ：[E_MAP_GLOBAL_SCAN_FAIL, E_MAP_NO_DETECTION, E_MAP_INSUFFICIENT_DETECTION, E_MAP_WALL_FRAME_FAIL, E_MAP_GRID_BUILD_FAIL, E_MAP_NO_NEW_WALL]
- ROS2 接口   ：（同进程类调用 WallMappingFSM）
- Context 读  ：wall_index, current_base_pose
- Context 写  ：current_wall_grid, new_wall_detected
- 日志要点    ：mapping 耗时、检测帧数、grid 完整度
- 备注        ：子 FSM 失败时把 error_code 透传到本层

#### CHECK_WALL_VALID
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：建网格完毕
- 退出条件    ：grid 合法 → NAVIGATE_TO_PHASE_WORKPOSE；new_wall=false → VERIFY_TASK_COMPLETE
- 成功跳转    ：NAVIGATE_TO_PHASE_WORKPOSE
- 失败跳转    ：WALL_ERROR_HANDLE / VERIFY_TASK_COMPLETE
- 超时        ：1s
- 可能异常码  ：[E_MAP_GRID_BUILD_FAIL]
- ROS2 接口   ：—
- Context 读  ：current_wall_grid, new_wall_detected
- Context 写  ：—
- 日志要点    ：grid 摘要：occupied slot 数、wall_frame
- 备注        ：grid 不完整但可用时（E_MAP_GRID_INCOMPLETE 仅 WARN）仍继续

#### NAVIGATE_TO_PHASE_WORKPOSE
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：初始进入左作业位，或 PairSelection 推荐切换到另一作业位
- 退出条件    ：导航 + 对墙微调成功 → RUN_PHASE_PERCEPTION
- 成功跳转    ：RUN_PHASE_PERCEPTION
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：60s（参数化）
- 重试上限    ：2
- 可能异常码  ：[E_NAV_*, E_NAV_FINE_ALIGN_FAIL, E_NAV_WORKPOSE_INVALID]
- ROS2 接口   ：Action client `/navigate_to_pose`，goal_type=LEFT_PHASE 或 RIGHT_PHASE，require_fine_alignment=true
- Context 读  ：current_phase, current_wall_grid.wall_frame_pose, config.left_phase_offset / right_phase_offset / desired_distance_to_wall
- Context 写  ：last_nav_result, current_base_pose
- 日志要点    ：alignment_error
- 备注        ：切换作业位后必须重新运行 RUN_PHASE_PERCEPTION，再重新选择 pair，不直接执行切换前的候选

#### RUN_PHASE_PERCEPTION
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：到达 phase 作业位
- 退出条件    ：子 FSM 返回 → RUN_PAIR_SELECTION
- 成功跳转    ：RUN_PAIR_SELECTION
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：20s（子 FSM 自带超时）
- 重试上限    ：2
- 可能异常码  ：[E_PERC_NO_LOCAL_DETECTION, E_PERC_ASSOCIATION_FAIL, E_PERC_LOCAL_SCAN_TIMEOUT]
- ROS2 接口   ：（同进程类调用 PhasePerceptionFSM）
- Context 读  ：current_phase, current_wall_grid, current_base_pose
- Context 写  ：visible_slot_ids, current_wall_grid 中对应 slot 的 latest_pose_robot
- 日志要点    ：可见 slot 数、关联成功率
- 备注        ：每次抓取前都要做一次新鲜局部感知

#### RUN_PAIR_SELECTION
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：局部感知完毕
- 退出条件    ：选出当前作业位 pair → DISPATCH_PAIR_GRASP；推荐另一作业位 → DECIDE_NEXT_PHASE；无安全候选 → WALL_ERROR_HANDLE
- 成功跳转    ：DISPATCH_PAIR_GRASP / DECIDE_NEXT_PHASE
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：2s
- 可能异常码  ：[E_PAIR_NO_CANDIDATE, E_PAIR_NO_REACHABLE, E_PAIR_DUAL_CONFLICT, E_PAIR_SINGLE_NOT_ALLOWED]
- ROS2 接口   ：（同进程类调用 PairSelectionFSM）
- Context 读  ：current_phase, current_wall_grid, visible_slot_ids, config.allow_single_arm_grasp, config.max_adjacent_column_height_delta
- Context 写  ：current_grasp_pair
- 日志要点    ：列高度、左右作业位安全候选数、最终 pair_id、左右手 slot_id
- 备注        ：PairSelection 同时评估左右作业位；作业位内只看局部上/左优先，全局只用于相邻列高度安全过滤

#### DISPATCH_PAIR_GRASP
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：选好 pair
- 退出条件    ：Action goal 发出 → WAIT_PAIR_GRASP_RESULT
- 成功跳转    ：WAIT_PAIR_GRASP_RESULT
- 失败跳转    ：WALL_ERROR_HANDLE
- 超时        ：2s（仅 send_goal 阶段）
- 可能异常码  ：[E_GRASP_GOAL_REJECTED, E_GRASP_INVALID_PAIR]
- ROS2 接口   ：Action client `/execute_pair_grasp`
- Context 读  ：current_grasp_pair, config.pair_grasp_timeout
- Context 写  ：current_grasp_goal_handle
- 日志要点    ：goal 发出、goal_handle 是否 accepted
- 备注        ：仅负责"投递"，等待结果在下个状态

#### WAIT_PAIR_GRASP_RESULT
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：goal accepted
- 退出条件    ：Action 完成 → UPDATE_GRID_AFTER_GRASP
- 成功跳转    ：UPDATE_GRID_AFTER_GRASP
- 失败跳转    ：UPDATE_GRID_AFTER_GRASP（失败结果也要更新 grid）
- 超时        ：120s（参数化，pair 抓取总时长）
- 可能异常码  ：[E_GRASP_GOAL_TIMEOUT, 透传 5xxx 抓取层错误码]
- ROS2 接口   ：Action client 接收 feedback + result
- Context 读  ：current_grasp_goal_handle
- Context 写  ：last_pair_grasp_result
- 日志要点    ：feedback 中 stage 变化、vacuum 峰值、execution_time
- 备注        ：成功失败都到 UPDATE_GRID_AFTER_GRASP；超时本身视为失败但要先 cancel goal

#### UPDATE_GRID_AFTER_GRASP
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：拿到 PairGraspResult
- 退出条件    ：grid 更新完毕 → DECIDE_NEXT_PAIR
- 成功跳转    ：DECIDE_NEXT_PAIR
- 失败跳转    ：WALL_ERROR_HANDLE（FATAL 级如掉箱）
- 超时        ：1s
- 可能异常码  ：[E_MOT_DROP_BOX, E_VAC_LOST_DURING_CARRY]（FATAL 时上报）
- ROS2 接口   ：pub `/fsm/grid_snapshot`
- Context 读  ：last_pair_grasp_result, current_grasp_pair
- Context 写  ：current_wall_grid 对应 slot 的 status (REMOVED/FAILED) + retry_count
- 日志要点    ：本 pair 的 slot 状态变化
- 备注        ：SUCCESS_BOTH → 两 slot=REMOVED；SUCCESS_LEFT_ONLY/RIGHT_ONLY → 仅成功侧 slot=REMOVED；失败侧 retry_count+1，超 max 则 BLOCKED

#### DECIDE_NEXT_PAIR
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：grid 已更新
- 退出条件    ：wall 仍有 OCCUPIED slot → RUN_PHASE_PERCEPTION；wall 无 OCCUPIED slot → DECIDE_NEXT_WALL
- 成功跳转    ：RUN_PHASE_PERCEPTION / DECIDE_NEXT_WALL
- 失败跳转    ：—
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：—
- Context 读  ：current_wall_grid, current_phase
- Context 写  ：—
- 日志要点    ：剩余 graspable 数
- 备注        ：不再以单个作业位覆盖范围作为完成边界，wall 级无 OCCUPIED 才进入完成确认

#### DECIDE_NEXT_PHASE
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：PairSelection 推荐 phase 与当前底盘作业位不同
- 退出条件    ：current_phase 切到推荐 phase → NAVIGATE_TO_PHASE_WORKPOSE
- 成功跳转    ：NAVIGATE_TO_PHASE_WORKPOSE
- 失败跳转    ：—
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：—
- Context 读  ：current_phase, recommended_phase, current_wall_grid
- Context 写  ：current_phase = recommended_phase
- 日志要点    ：phase 切换事件
- 备注        ：切换后必须重新局部感知，避免用旧作业位视角的可见性直接执行抓取

#### DECIDE_NEXT_WALL
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：当前 wall 完成
- 退出条件    ：wall_index 增加 → NAVIGATE_TO_OBSERVATION_POSE；达到 max_walls 限制 → VERIFY_TASK_COMPLETE
- 成功跳转    ：NAVIGATE_TO_OBSERVATION_POSE / VERIFY_TASK_COMPLETE
- 失败跳转    ：—
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：—
- Context 读  ：wall_index, config.max_walls
- Context 写  ：wall_index += 1, current_phase = LEFT_PHASE, empty_confirm_count = 0
- 日志要点    ：wall 切换
- 备注        ：—

#### VERIFY_TASK_COMPLETE

- 进入条件    ：候选完成
- 退出条件    ：多帧无箱确认 → WALL_DONE；仍看到 → RUN_WALL_MAPPING（重扫确认）
- 成功跳转    ：WALL_DONE
- 失败跳转    ：RUN_WALL_MAPPING / WALL_ERROR_HANDLE
- 超时        ：30s（多帧时间窗）
- 可能异常码  ：[E_WALL_TASK_COMPLETE_TIMEOUT, E_WALL_EMPTY_VERIFY_CONFLICT]
- ROS2 接口   ：sub `/perception/box_detections`（开窗 N 帧）
- Context 读  ：config.empty_confirm_frames
- Context 写  ：empty_confirm_count
- 日志要点    ：连续 N 帧 detection 数
- 备注        ：N 帧全部 detection_count=0 才算完成；任一帧有箱→重扫

#### WALL_DONE

- 退出条件    ：Action result 返回（终态）
- 成功跳转    ：（终态）
- 失败跳转    ：—
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：Action server 发送 result
- Context 读  ：wall_index, total counters
- Context 写  ：—
- 日志要点    ：task summary
- 备注        ：返回 Action result.success=true

#### WALL_ERROR_HANDLE
- 所属 FSM    ：WallDestackingFSM
- 进入条件    ：任何状态返回 FAILURE 且 level≥ERROR
- 退出条件    ：WallRecoveryFSM 决策完毕 → 跳到推荐目标状态 或 ABORT
- 成功跳转    ：动态（由 WallRecoveryFSM 决定）
- 失败跳转    ：Action result.success=false（终态）
- 超时        ：30s
- 可能异常码  ：透传
- ROS2 接口   ：（同进程类调用 WallRecoveryFSM）
- Context 读  ：last_error
- Context 写  ：清除 last_error 或保留
- 日志要点    ：recovery action 选择
- 备注        ：见 §8 WallRecoveryFSM 内部状态

## 4. WallMappingFSM（你负责，子 FSM）

> 所在节点：`wall_destacking_strategy_node`（同进程类调用）。从全局观察位扫描当前最外侧货箱墙、估计 wall_frame、生成 5×5 逻辑网格。

### 4.1 状态总览

```text
START_WINDOW → COLLECT_FRAMES → FILTER_DETECTIONS → ESTIMATE_WALL_FRAME
  → BUILD_5X5_GRID → INIT_GRID_SLOTS → VALIDATE_GRID → CHECK_NEW_WALL → REPORT
异常：MAPPING_ERROR
```

### 4.2 状态卡

#### START_WINDOW
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：父 FSM 调用 reset+tick
- 退出条件    ：开窗参数就绪 → COLLECT_FRAMES
- 成功跳转    ：COLLECT_FRAMES
- 失败跳转    ：MAPPING_ERROR
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：—
- Context 读  ：config.mapping_window_frames, config.mapping_window_sec
- Context 写  ：mapping_buffer = []
- 日志要点    ：开始时间
- 备注        ：本子 FSM 不主动 trigger 视觉，仅消费现有连续 Topic

#### COLLECT_FRAMES
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：开窗开始
- 退出条件    ：达到 N 帧或 T 秒 → FILTER_DETECTIONS
- 成功跳转    ：FILTER_DETECTIONS
- 失败跳转    ：MAPPING_ERROR
- 超时        ：config.mapping_window_sec（默认 3s）
- 重试上限    ：1
- 可能异常码  ：[E_MAP_NO_DETECTION, E_PERC_LOCAL_SCAN_TIMEOUT]（虽然名带 LOCAL，全局也复用）
- ROS2 接口   ：sub `/perception/box_detections`
- Context 读  ：mapping_buffer
- Context 写  ：mapping_buffer.append(每帧)
- 日志要点    ：每帧 detection 数、frame_seq
- 备注        ：要求帧间不重复（按 frame_seq 去重）；若 T 秒内连续 ≥3 帧空 → 提前判 E_MAP_NO_DETECTION

#### FILTER_DETECTIONS
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：开窗结束
- 退出条件    ：滤波后 ≥ min_valid 个检测 → ESTIMATE_WALL_FRAME；不足 → MAPPING_ERROR
- 成功跳转    ：ESTIMATE_WALL_FRAME
- 失败跳转    ：MAPPING_ERROR
- 超时        ：2s
- 可能异常码  ：[E_MAP_INSUFFICIENT_DETECTION]
- ROS2 接口   ：—
- Context 读  ：mapping_buffer, config.min_valid_detections (默认 8)
- Context 写  ：filtered_detections
- 日志要点    ：过滤前/后数量
- 备注        ：滤除 confidence 低、size 偏离 0.4±0.1、frame_id 不对的检测

#### ESTIMATE_WALL_FRAME
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：有效检测足够
- 退出条件    ：拟合 wall 平面成功 → BUILD_5X5_GRID
- 成功跳转    ：BUILD_5X5_GRID
- 失败跳转    ：MAPPING_ERROR
- 超时        ：3s
- 可能异常码  ：[E_MAP_WALL_FRAME_FAIL, E_MAP_WALL_FRAME_LOW_CONFIDENCE]
- ROS2 接口   ：无；直接使用 `perception_adapter_node` 已转换到 `base_link` 的 BoxDetectionArray
- Context 读  ：filtered_detections
- Context 写  ：wall_frame_pose_robot, wall_estimation_confidence
- 日志要点    ：平面拟合残差、置信度
- 备注        ：用 adapter 输出的箱面中心 / 法向做 RANSAC 拟合 wall 平面；origin 取检测点云重心投影；X 法向朝相机；Y 朝左

#### BUILD_5X5_GRID
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：wall_frame 就绪
- 退出条件    ：25 个 slot 的 expected_pose 计算完毕 → INIT_GRID_SLOTS
- 成功跳转    ：INIT_GRID_SLOTS
- 失败跳转    ：MAPPING_ERROR
- 超时        ：1s
- 可能异常码  ：[E_MAP_GRID_BUILD_FAIL]
- ROS2 接口   ：—
- Context 读  ：wall_frame_pose_robot, config.box_size, config.grid_shape
- Context 写  ：current_wall_grid（25 slot 的 expected_pose_robot）
- 日志要点    ：grid 尺寸
- 备注        ：按 `business.box_size`（默认 0.4m）+ wall_frame 构造 25 个理论中心点

#### INIT_GRID_SLOTS
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：grid 几何就绪
- 退出条件    ：每个 slot 与最近的检测做匹配 → VALIDATE_GRID
- 成功跳转    ：VALIDATE_GRID
- 失败跳转    ：MAPPING_ERROR
- 超时        ：2s
- 可能异常码  ：[E_MAP_GRID_INCOMPLETE]（仅 WARN，不阻塞）
- ROS2 接口   ：—
- Context 读  ：current_wall_grid, filtered_detections, config.match_tolerance_y, config.match_tolerance_z
- Context 写  ：每个 slot 的 latest_pose_robot / visible / confidence
- 日志要点    ：匹配成功 slot 数
- 备注        ：匹配阈值 Y±15cm、Z±15cm；匹配上则 status=OCCUPIED；匹配不上但理论存在 → 仍 OCCUPIED + visible=false

#### VALIDATE_GRID
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：slot 初始化完毕
- 退出条件    ：grid 合法 → CHECK_NEW_WALL
- 成功跳转    ：CHECK_NEW_WALL
- 失败跳转    ：MAPPING_ERROR
- 超时        ：1s
- 可能异常码  ：[E_MAP_GRID_BUILD_FAIL]
- ROS2 接口   ：—
- Context 读  ：current_wall_grid
- Context 写  ：—
- 日志要点    ：occupied/unknown 数量
- 备注    ：要求至少 row0 全 5 个 + col0 全 5 个有匹配；V1 严格

#### CHECK_NEW_WALL
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：grid 合法
- 退出条件    ：判定是否新墙 → REPORT
- 成功跳转    ：REPORT
- 失败跳转    ：MAPPING_ERROR
- 超时        ：1s
- 可能异常码  ：[E_MAP_NO_NEW_WALL]（INFO 级，正常）
- ROS2 接口   ：—
- Context 读  ：current_wall_grid, wall_estimation_confidence
- Context 写  ：new_wall_detected
- 日志要点    ：是否新墙
- 备注    ：V1 简单判定：occupied>=15 且置信度>=0.6 即新墙；否则 new_wall_detected=false

#### REPORT
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：判定完成
- 退出条件    ：父 FSM 取走结果 → 终态
- 成功跳转    ：（终态 SUCCESS）
- 失败跳转    ：—
- 超时        ：1s
- 可能异常码  ：[]
- ROS2 接口   ：pub `/fsm/grid_snapshot`
- Context 读  ：current_wall_grid, new_wall_detected
- Context 写  ：—
- 日志要点    ：完整 grid summary
- 备注    ：—

#### MAPPING_ERROR
- 所属 FSM    ：WallMappingFSM
- 进入条件    ：任何子状态失败
- 退出条件    ：携带 error_code 返回父 FSM → 终态 FAILURE
- 成功跳转    ：—
- 失败跳转    ：（终态 FAILURE）
- 超时        ：1s
- 可能异常码  ：透传
- ROS2 接口   ：pub `/fsm/error`
- Context 读  ：last_error
- Context 写  ：—
- 日志要点    ：完整错误
- 备注    ：—

## 8. BaseNavigationFSM（工程师 A，精简卡 + FINE_ALIGN 详卡）

> 所在节点：`navigation_manager_node`。负责所有底盘移动 + 对墙微调 + 定位健康监控 + clear_error 第 1/3/4 阶段的实际 ROS 调用。**本节点是 Nav2 / AMCL / chassis_node / twist_mux 的唯一适配层**。本期 A 参考既有自研导航代码（见 `codes/IK、视觉雷达感知与底盘导航代码导览.md`）补充内部实现。

### 8.1 接口契约（必须遵守）

```text
对策略层入口（唯一）：NavigateToPose.action（Goal/Result/Feedback 见 03 §2.3.2）
对策略层心跳：/fsm/nav_health (std_msgs/Bool)，1Hz

关键 Feedback：current_state 字符串、distance_remaining、alignment_error_current
关键 Result：success, actual_base_pose, position_error, yaw_error, alignment_error,
            workpose_valid, error_code, failure_reason
```

下游 ROS 接口（详见 03 §3.3）：

- Nav2：`NavigateToPose` Action（adapter 转发）、`/lifecycle_manager_*/is_active`、`/local_costmap/clear_entirely_local_costmap`、`/global_costmap/clear_entirely_global_costmap`
- AMCL：`/initialpose` (Pub)、`/amcl_pose` (Sub)
- chassis_node：`/chassis_node/{enable, disable, reset_fault, status}`
- twist_mux：`/cmd_vel_align` (Pub，FINE_ALIGN 专用)、`/estop` (Pub，clear_error 第 1 阶段)
- 视觉反馈（FINE_ALIGN 用）：`/perception/box_detections` (Sub)，由 `perception_adapter_node` 提供

### 8.2 必须实现的对外语义

无论内部状态如何，A 必须保证：

1. goal_type=OBSERVATION 时不做 fine_alignment
2. goal_type=LEFT_PHASE/RIGHT_PHASE 且 require_fine_alignment=true 时必须进 FINE_ALIGN
3. 失败 error_code 必须从 04 文档 4xxx 段中选
4. cancel 时必须在 2s 内停下并返回 result（success=false, error_code=E_NAV_GOAL_CANCELLED）
5. 急停信号（`/safety/estop=true`）时必须立刻停下并 cancel 当前 goal
6. **不直接发 `/cmd_vel` 或 `/cmd_vel_nav`**：导航走 Nav2 Action（Nav2 内部发 `/cmd_vel_nav`），FINE_ALIGN 走 `/cmd_vel_align`
7. clear_error 协议（02 §1.3.x）的第 1/3/4 阶段由本节点实际执行，通过 `/nav/base_recovery` 内部服务暴露给 task_manager 调

### 8.3 推荐内部状态骨架

```text
RECEIVE_GOAL → CHECK_PRECONDITION → CHECK_LOCALIZATION
  → PLAN_PATH → EXECUTE → MONITOR → [FINE_ALIGN] → VERIFY → REPORT
异常：NAV_ERROR

并行 / 后台监控（独立 callback group，非主 tick 链）：
  - LOCALIZATION_WATCH：订 /amcl_pose，监控协方差对角线
  - CHASSIS_WATCH：订 /chassis_node/status，监控 fault_a/b、heartbeat_ok
  - LIFECYCLE_WATCH：定期调 /lifecycle_manager_*/is_active
```

**A 实现要点**：

- 至少补全：`RECEIVE_GOAL` / `CHECK_LOCALIZATION` / `EXECUTE` / `FINE_ALIGN` / `VERIFY` / `NAV_ERROR` 六张完整规格卡（FINE_ALIGN 见下）
- 每个状态超时 / error_code / Context 字段由 A 在 PR 中补全
- `LOCALIZATION_WATCH`：协方差对角线（xx、yy、yaw）超阈值（默认 xx/yy > 1.0、yaw > 0.5）持续 2s → raise `E_NAV_LOCALIZATION_LOST` 给当前 goal，自动 cancel goal 并返回 result.error_code

### 8.4 FINE_ALIGN 详细规格卡（**新增，P0**）

#### FINE_ALIGN
- 所属 FSM    ：BaseNavigationFSM
- 进入条件    ：EXECUTE 完成且 `Goal.require_fine_alignment=true` 且 `Goal.goal_type ∈ {LEFT_PHASE, RIGHT_PHASE}`
- 退出条件    ：连续 N 帧（默认 5）法向偏差 < `align_yaw_tol`（0.05rad）且距离误差 < `align_dist_tol`（0.02m） → VERIFY；超时或失败 → NAV_ERROR
- 成功跳转    ：VERIFY
- 失败跳转    ：NAV_ERROR
- 超时        ：15s（参数化 `fine_align_timeout`）
- 重试上限    ：1（超时后清局部 costmap 重试一次；再失败彻底 NAV_ERROR）
- 可能异常码  ：[E_NAV_FINE_ALIGN_FAIL, E_NAV_FINE_ALIGN_NO_FEEDBACK, E_PERC_LOCAL_SCAN_TIMEOUT]
- ROS2 接口   ：
  - Sub `/perception/box_detections`（取当前 phase 内、视野中央的几个 box 法向均值）
  - Pub `/cmd_vel_align` (`geometry_msgs/Twist`)，**仅 `linear.x` 与 `angular.z`**，绝对值上限：`|linear.x| ≤ 0.05 m/s`、`|angular.z| ≤ 0.2 rad/s`
  - Sub `/amcl_pose`（用于位置反馈）
  - Sub `/chassis_node/status`（fault 时立即 abort）
- Context 读  ：goal.desired_distance_to_wall, goal.desired_yaw_to_wall, goal.phase, latest_box_detections
- Context 写  ：alignment_error_current（每 tick 更新）、align_consec_pass_count
- 日志要点    ：每 200ms 一条 metric：`{yaw_err, dist_err, vel_x, vel_z, det_count}`
- 备注        ：
  - 进入 FINE_ALIGN 前必须确保 Nav2 coarse goal 已成功结束；若是在粗定位容差内提前切入，必须先 cancel Nav2 goal 并等待 cancel ack，避免 `/cmd_vel_nav` 与 `/cmd_vel_align` 长时间竞争
  - 进入时停发 align 速度也是合法的：如果一进来误差就已达标，连续 N 帧通过即可直接 VERIFY
  - 退出（无论成功失败）必须显式发一帧 `Twist{linear.x=0, angular.z=0}` 到 `/cmd_vel_align` 然后停发；不停发会被 twist_mux 在 0.5s 后自动 timeout
  - FINE_ALIGN 失败后如需兜底收尾，必须重新发送新的 Nav2 goal，不依赖旧 goal 继续执行
  - V1 反馈源：只用视觉法向；激光最近点反馈作为 V2 改进项
  - 若 `/perception/box_detections` 在 1s 内没有当前 phase 对应的 box，raise `E_NAV_FINE_ALIGN_NO_FEEDBACK`

### 8.5 状态广播

A 必须通过 Action Feedback 的 `current_state` 字段告诉策略层当前内部状态名，便于 UI 展示。FINE_ALIGN 期间还要持续填 `alignment_error_current`（其他状态填 NaN）。

### 8.6 LOCALIZATION_WATCH 详细行为

后台监控线程，独立 callback group：

- 订 `/amcl_pose`（`PoseWithCovarianceStamped`），10Hz
- 取协方差对角线 `cov_xx = covariance[0]`、`cov_yy = covariance[7]`、`cov_yaw = covariance[35]`
- 当 `cov_xx > 1.0 || cov_yy > 1.0 || cov_yaw > 0.5` 持续 2s：
  1. 当前若有 active Nav2 goal，cancel 之
  2. raise `E_NAV_LOCALIZATION_LOST` 给当前 Action 客户端（return result.error_code）
  3. `/fsm/nav_health` 翻 False
- 当协方差降回阈值内连续 2s：`/fsm/nav_health` 翻 True
- 重定位接口：当 task_manager 调本节点的内部 method `relocate_with_pose(pose)` 时，向 `/initialpose` publish 一次（连发 3 次，间隔 200ms，covariance 写 `[0.25, 0.25, ..., 0.1]`），不依赖 `lidar_init_pose_node`

### 8.7 既有接入点

详见 `codes/IK、视觉雷达感知与底盘导航代码导览.md` §6 / §7。A 实现的本质是把那一节里的所有"FSM 缺口"在本节点内填上：

- Nav2 `NavigateToPose` Action 转发
- 两个 lifecycle manager 的 is_active 探针
- `/initialpose` 重发（自管 `sent_` 状态，可重发）
- `/amcl_pose` 协方差监控（AMCL 不自带 lost topic）
- 直接调 costmap 的 clear service（behavior_server 没有 clear_costmap plugin）
- `/save_map` 发完后 sleep + stat 文件（mapper_node 无 ack）
- `/chassis_node/{enable, reset_fault}` 调用
- `/cmd_vel_align` 通道使用（**前提是 twist_mux.yaml 已扩展，见 03 §10**）

## 9. PairGraspExecutionFSM（工程师 B，精简卡 + CHECK_VACUUM 详卡）

> 所在节点：`pair_grasp_execution_node`。负责双臂抓取、吸附、搬运、释放。**本节点是 MoveIt / IK / 机械臂控制 / 吸盘 IO 的唯一适配层**，FSM 策略层不直接加载 IK 插件。本期 B 参考既有 IK/规划代码（见 `codes/IK、视觉雷达感知与底盘导航代码导览.md` §3）补充内部实现。
>
> 吸盘 IO 拆出独立 `vacuum_io_node`（详见 03 §3.4a），本节点订其 `/vacuum/pressure_raw` 并发 `/vacuum/cmd`。这样 mock first：`vacuum_io_node` 先用 mock 实现，硬件后端后接，本节点不动。

### 9.1 接口契约（必须遵守）

```text
对策略层入口（唯一）：ExecutePairGrasp.action（Goal/Result/Feedback 见 03 §2.3.3）

关键 Feedback：current_state, current_stage, progress_percent, vacuum_left_kpa, vacuum_right_kpa
关键 Result：success, result.result_code, result.left_result, result.right_result, result.failed_stage, result.error_code
```

下游 ROS 接口（详见 03 §3.4）：

- MoveIt：`MoveGroup` Action（adapter 内部调；策略层不直接调）、可选 `/compute_ik` Service
- ros2_control：`/dual_v5_arm_controller/follow_joint_trajectory` Action、`/torso_controller/follow_joint_trajectory` Action
- `/joint_states` (Sub)、`/alfa_collision_markers` (Sub，可选)
- 吸盘：`/vacuum/cmd` (Pub) → `vacuum_io_node`、`/vacuum/pressure_raw` (Sub) ← `vacuum_io_node`
- 自己转发：`/vacuum/pressure` (Pub) 给 UI

### 9.2 必须实现的对外语义

1. result_code 必须从 ResultCode 常量集（03 §7）中取，不允许自定义
2. error_code 必须从 04 文档 5xxx/6xxx 段中选
3. 任何阶段失败都必须返回 Action Result（success=false + PairGraspResult 的 result_code / failed_stage / error_code），不允许只返回 abort
4. dry_run=true 时只走规划路径，不真正运动，**也不发 `/vacuum/cmd`**；最终 result_code 仍按规划结果填
5. cancel 时必须确保机械臂在安全位停下；是否释放吸盘按 `business.vacuum.release_on_cancel` 执行
6. 急停信号时立即停下；默认保持吸盘（`business.vacuum.hold_on_estop=true`），避免未知姿态下直接掉箱
7. SUCCESS_BOTH 要求两手都真正放置成功，掉箱必须返回 FATAL（E_MOT_DROP_BOX）
8. **不直接加载 IK 插件**：用 `MoveGroupInterface::setFromIK()` 或进程内 `IkSolver`（来自 `scripts/ik_benchmark`）；详见 07 §3.5
9. **不调 `dual_arm_planner_node/~/plan_and_execute`**——那是无参数 demo，不能用作生产接口（07 §3.3）

### 9.3 推荐内部状态骨架

```text
RECEIVE_PAIR → CHECK_PAIR_VALID → PLAN_PREGRASP → MOVE_TO_PREGRASP
  → APPROACH_AND_CONTACT → CHECK_VACUUM → ATTACH_BOX_MODEL
  → PLAN_EXTRACT → EXECUTE_EXTRACT → PLAN_CARRY → EXECUTE_CARRY
  → RELEASE_BOX → RETREAT_SAFE → REPORT
异常：EXECUTION_ERROR
```

可进一步拆 4 个内部子 FSM：GraspPlanningFSM / VacuumContactFSM / CarryPlaceFSM / ExecutionRecoveryFSM。

**B 实现要点**：

- 至少补全：`RECEIVE_PAIR` / `PLAN_PREGRASP` / `APPROACH_AND_CONTACT` / `CHECK_VACUUM`（见 §9.4）/ `EXECUTE_CARRY` / `RELEASE_BOX` / `EXECUTION_ERROR` 七张完整规格卡
- 关键 metric 必须记日志：plan_time、move_time、vacuum_buildup_time、execution_total_time
- 真空压力 `/vacuum/pressure_raw` 来自 `vacuum_io_node`，必须 ≥20Hz
- IK 性能不达标时切换求解器：先 KDL（默认），需要双臂协调时切 bio_ik（kinematics.yaml 已配）；切换要在节点构造期完成，**不要在抓取过程中切**

### 9.4 CHECK_VACUUM 详细规格卡（**新增，P0**）

#### CHECK_VACUUM
- 所属 FSM    ：PairGraspExecutionFSM
- 进入条件    ：APPROACH_AND_CONTACT 完成（双臂吸盘已贴近箱面）
- 退出条件    ：双臂真空压力均 ≤ `vacuum_attach_threshold_kpa`（默认 -50.0，越负越好）持续 `vacuum_hold_ms`（默认 150ms） → ATTACH_BOX_MODEL；超时或一侧未达标 → 见"失败分支"
- 成功跳转    ：ATTACH_BOX_MODEL
- 失败跳转    ：EXECUTION_ERROR；单边失败时 result_code=SUCCESS_LEFT_ONLY / SUCCESS_RIGHT_ONLY（根据 grasp_mode 决定是否允许）
- 超时        ：`vacuum_buildup_timeout`（默认 800ms）
- 重试上限    ：1（先 RELEASE → 再做一次 APPROACH_AND_CONTACT 微调；再失败彻底放弃）
- 可能异常码  ：[E_VAC_NOT_REACHED, E_VAC_UNILATERAL_FAIL, E_VAC_SENSOR_OFFLINE, E_VAC_TIMEOUT]
- ROS2 接口   ：
  - Pub `/vacuum/cmd` (`fsm_msgs/VacuumCommand`)：`left_on=true, right_on=true` 开启双臂吸盘
  - Sub `/vacuum/pressure_raw`（`std_msgs/Float32MultiArray`）：每 50ms 一帧
  - Pub `/vacuum/pressure`（转发 + 标注 timestamp）给 UI
  - Sub `/vacuum/health` 判 sensor 在线
- Context 读  ：grasp_pair.grasp_mode, config.vacuum_attach_threshold_kpa, config.vacuum_buildup_timeout
- Context 写  ：last_vacuum_left_kpa, last_vacuum_right_kpa, vacuum_buildup_time_ms
- 日志要点    ：进入瞬间记录"vacuum_on_cmd"；每 50ms 一条 metric `{left_kpa, right_kpa}`；退出时记录 `vacuum_buildup_time_ms` 与最终判定
- 备注        ：
  - **判定阈值是负数**：行业惯例真空表读数越负越好；adapter 内部 abs() 或符号比较都行，但 yaml 默认值要标注清楚单位与符号
  - sensor 在 200ms 内无新数据 → 立即判 `E_VAC_SENSOR_OFFLINE`，不要等到 buildup_timeout
  - 单边失败的处理：`grasp_mode == DUAL` 时进 EXECUTION_ERROR；`grasp_mode == LEFT_ONLY/RIGHT_ONLY` 且失败那侧本来就不该吸 → 不算失败（adapter 仅判活的那一侧）
  - 退出时**不要关吸盘**——下一状态 ATTACH_BOX_MODEL 假定吸盘持续吸；只有 RELEASE_BOX、可安全释放的 EXECUTION_ERROR、或配置允许的 cancel 才发 `VacuumCommand{left_on=false,right_on=false}`。急停默认保持吸盘，由人工确认后释放。

### 9.5 状态广播

B 必须通过 Action Feedback 持续报告 current_state / current_stage / vacuum_left_kpa / vacuum_right_kpa，便于策略层做监控和 UI 展示。

### 9.6 既有接入点

- 既有 IK 代码：通过 `MoveGroupInterface` + `kinematics.yaml`，或 `IkSolver` 进程内库（07 §3.4）
- 既有轨迹规划代码：MoveIt `MoveGroup` Action
- 既有机械臂控制代码：`/dual_v5_arm_controller/follow_joint_trajectory`、`/torso_controller/follow_joint_trajectory`
- 吸盘 / 真空硬件：**通过 `vacuum_io_node`** 间接对接，**不直接** open CAN / Modbus / GPIO
- 真空压力传感器：同上，封装在 `vacuum_io_node`

## 10. SafetyMonitorFSM（你负责）

> 所在节点：`safety_monitor_node`。监听急停、安全区、通信看门狗。

### 10.1 状态总览

很简单，只有 3 个状态：

```text
NORMAL ⇄ WARNING ⇄ EMERGENCY
```

### 10.2 状态卡

#### NORMAL
- 所属 FSM    ：SafetyMonitorFSM
- 进入条件    ：节点启动 / 故障解除
- 退出条件    ：检测到 WARNING 级 → WARNING；检测到 ESTOP 级 → EMERGENCY
- 成功跳转    ：—
- 失败跳转    ：WARNING / EMERGENCY
- 超时        ：无
- 可能异常码  ：[]
- ROS2 接口   ：pub `/safety/status` 10Hz，`/safety/estop`=false
- Context 读  ：last_heartbeats, last_estop_signal
- Context 写  ：safety_status
- 日志要点    ：进入心跳 1Hz
- 备注    ：—

#### WARNING
- 所属 FSM    ：SafetyMonitorFSM
- 进入条件    ：通信丢失 / 安全区 violation / 碰撞风险
- 退出条件    ：原因解除 → NORMAL；升级为 ESTOP → EMERGENCY
- 成功跳转    ：NORMAL
- 失败跳转    ：EMERGENCY
- 超时        ：30s（持续未解除升级 EMERGENCY）
- 可能异常码  ：[E_SAFETY_ZONE_VIOLATED, E_SAFETY_COMM_LOST, E_SAFETY_COLLISION_RISK]
- ROS2 接口   ：pub `/safety/status` 含 details
- Context 读  ：—
- Context 写  ：safety_status
- 日志要点    ：触发原因
- 备注    ：本状态不主动急停，让 FSM 各层自己决策

#### EMERGENCY
- 所属 FSM    ：SafetyMonitorFSM
- 进入条件    ：硬件急停 / 软件急停 / 通信全失 / 看门狗超时
- 退出条件    ：硬件急停输入 `/estop_button=False` 且软件急停 `/safety/sw_estop` 已 release，随后 `/clear_error` 五阶段全成 → NORMAL
- 成功跳转    ：NORMAL
- 失败跳转    ：—（停在 EMERGENCY，保持广播）
- 超时        ：无
- 可能异常码  ：[E_SAFETY_ESTOP_HW, E_SAFETY_ESTOP_SW, E_SAFETY_COMM_LOST, E_SAFETY_WATCHDOG]
- ROS2 接口   ：
  - Pub `/safety/status`（`estop=true`）+ `/safety/estop=true` (latched)
  - 急停源仍 active 时持续 Pub `/estop=true`（独立于硬件按钮，软件强制锁 twist_mux）
  - 急停源解除后停止发 `/estop=true`，但不主动发 False；由 task_manager 调 `/clear_error` 走五阶段（02 §1.3.x），第 1 阶段通过 `/nav/base_recovery` 发 `/estop=false` 解锁
  - Sub `/estop_button`（硬件 / mock 按钮输入）
  - Service `/safety/sw_estop` (`std_srvs/SetBool`)：`data=true` 置软件急停源 active，`data=false` 释放软件急停源；UI 不允许直接 publish `/estop`
- Context 读  ：—
- Context 写  ：safety_status
- 日志要点    ：急停源、入口时刻、五阶段每步进度
- 备注        ：
  - 进入即广播 estop，所有 FSM 节点订阅后 abort
  - **退出严格走五阶段**，不允许"按一下复位按钮就跳回 NORMAL"——否则会出现"twist_mux 锁未释放但 FSM 已 NORMAL"的不一致状态
